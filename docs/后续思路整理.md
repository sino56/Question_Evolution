## 2. 进化机制

### 2.1 当前失败退回机制

| 当前状态 | 处理动作 |
| --- | --- |
| 生成题不合法 | 回到父题 |
| 进化后分数反而升高 | 回到父题，切换 operator |
| 题目合法但压测效果不清晰 | 留在当前子题，切换 operator |
| 进化形成有效边界 | 保留当前题并停止继续进化 |

### 2.2 更合理的回退策略

1. **单次失败**：回到直接父题并换算子，保持当前机制。
2. **同一父题连续两个或多个算子失败**：回到最近一个"已验证有效"的祖先题。
3. **连续漂移、得分异常或无可信祖先**：再回到最初的原始题。
4. **持久化信息**：保存 `root_prompt`、完整 `ancestor_history`、每个版本的分数和效果标签，而不是只有 `prompt_old`。
5. **回滚目标**：回滚目标应是 `best_valid_ancestor`，不应简单固定为父题或原始题。

> 默认回直接父题，连续失败时逐级回退到最佳有效祖先，最后才回原始题。

### 2.3 多层回溯

建议机制：沿祖先链的多层回溯，例如 `Q3 → Q2 → Q1 → Q0`，根据历史效果选择最近的可靠节点。

```text
Q0 原始题
└─ Q1 有效
    └─ Q2 效果不明确
        └─ Q3 失败
```

合理处理：

1. Q3 失败，先回到直接父题 Q2。
2. 如果从 Q2 换多个算子仍失败，继续回到 Q1。
3. Q1 是已验证有效节点，就从 Q1 重新选择算子。
4. 只有所有中间节点都不可信时，才退回原始题 Q0。

这叫**祖先链多层回溯**。如果从 Q1 换不同算子生成 `Q2-A`、`Q2-B`、`Q2-C` 并比较这些分支，则进一步变成了**树搜索**。

## 3. 算子进化机制

### 3.1 核心参数

考虑稳定性，一个算子可以生成 `K` 道题，然后每道题评分 `R` 次，以保证得分的稳定性。另外可以对比：

- 一个算子生成的 `K` 道题之间的差异性；
- 不同样本利用同一个算子的差异性。

两个参数：

- **K**：一个算子针对同一道父题生成多少道候选题。
- **R**：每道候选题独立回答并评分多少次。

### 3.2 核心价值

1. **评分稳定性**：通过 `R` 次评分获得均值、方差和置信区间，避免一次模型波动决定进化结果。
2. **算子有效性**：观察一个算子生成的 `K` 道题，有多少能稳定降低得分，而不是只看最优的一道。
3. **算子内差异性**：判断算子是否真正生成不同的问题，还是只进行表面措辞变化。
4. **跨样本泛化性**：判断同一算子是否只对某类题有效，还是能稳定作用于多个题型。

### 3.3 需要特别注意

- 重复评分必须包含重新生成答案，不能只对同一答案重复调用评分模型，否则只能测评审波动。
- 如果模型、温度、seed 和 prompt 完全一致，多次结果可能高度相关，`R` 次不等于 `R` 个独立样本。
- 父题也要用相同条件评分 `R` 次，比较候选题与父题的分数分布，而不是拿候选均值和父题的一次历史得分比较。
- 不能从 `K` 道题中直接挑最低分题作为算子成果，这会产生"赢家诅咒"：K 越大，越容易随机出现一个异常低分题。
- 低分不一定代表有效进化，还可能来自不可回答、信息缺失、格式陷阱或评分标准漂移，复杂度和可回答性校验仍然必须保留。
- 跨样本比较时需要按原始难度、题型和失败模式分组，否则样本差异会被错误归因给算子。

### 3.4 建议指标

#### 每个候选题记录

- `score_mean`
- `score_std`
- `score_min` / `score_max`
- `score_confidence_interval`
- `score_drop_mean`
- `score_drop_probability`
- `validation_pass_rate`

#### 每个算子记录

- `candidate_valid_rate`
- `effective_candidate_rate`
- `median_score_drop`
- `stable_effective_rate`
- `within_sample_diversity`
- `cross_sample_effect_consistency`
- `failure_mode_distribution`

其中算子是否有效，建议判断为：

```text
有效率 = 稳定降低得分且校验合格的候选数 / K
```

而不是：**最低分候选是否比父题低**。

### 3.5 差异性评估

#### 同一父题下的 K 道题

应比较：

- 核心判断维度是否不同；
- 证据结构和约束组合是否不同；
- 预期触发的错误模式是否不同；
- 是否只是实体、数字或措辞替换；
- 是否仍然符合该算子的设计目标。

#### 不同样本使用同一算子时

应比较：

- 算子是否产生固定模板；
- 是否适配不同题型，而不是强行套同一种结构；
- 对不同样本类别的有效率和降分幅度；
- 失败是否集中在特定题型或特定原始分数区间。

> 差异性不能只用文本相似度。文本差异很大也可能考察同一能力；文本相似也可能存在关键逻辑差异。建议结合语义向量、结构化特征和 LLM 判定。

### 3.6 对当前流程的影响

当前 V3 在正式回答和评分前，就通过 `candidate_selection.py` 选出一道候选题。要实现这个设想，需要调整为：

```text
算子生成 K 道题
→ 校验并去重
→ 保留 M 道有效且有差异的题
→ 每道题回答并评分 R 次
→ 汇总稳定性和算子效果
→ 选择主链题目
→ 更新状态和算子 memory
```

也就是说，最终候选选择需要移动到评分之后，或者拆成两级：

- **预选择**：按有效性、可回答性和差异性将 K 道缩减到 M 道。
- **最终选择**：评分完成后，按稳定降分、质量和风险选择主链题目。

### 3.7 成本控制

完整执行成本约为：

```text
样本数 × 算子数 × K × R ×（回答成本 + 评分成本）
```

建议采用漏斗式策略：

1. 每个算子先生成 `K=3~5` 道题。
2. 校验和去重后保留 `M=2~3` 道。
3. 每道先评分一次。
4. 只有接近决策阈值、方差较大或可能有效的候选，追加到 `R=3~5` 次。
5. 最终选中的候选再做一次独立确认评分。

### 3.8 最终判断

这个方案值得实施，但核心不只是"多生成、多评分"，而是把评估单位从单道候选题提升为一次算子批次实验。最合理的决策依据应同时包含：

```text
算子有效率 + 得分下降稳定性 + 候选差异性 + 校验通过率 + 跨样本泛化性
```

这样才能区分"算子确实有效"和"多生成几次后偶然撞到一道低分题"。

## 4. 算子叠加

### 4.1 当前机制

当前机制实际是：

```text
一个样本
→ 路由得到 primary_operator + backup_operators
→ 每个候选题只使用一个 operator 生成
→ 多个算子生成并列候选
→ candidate_selection 选择一个候选
→ 效果归因给该候选对应的单一算子
```

因此，当前属于：

- **支持算子并行竞争**：O10 生成候选 A、O12 生成候选 B，再选择一个。
- **不支持算子组合生成**：没有在一次生成中执行 `O10 + O12`。
- **不支持显式算子链**：没有 `operator_sequence = [O10, O12]` 或各阶段结果记录。
- **效果只归因给一个算子**：`analyze_evolution_effect.py` 读取的是单个 `selected_operator`。

### 4.2 跨轮次的隐式叠加

```text
Q0 --O10--> Q1
Q1 --O12--> Q2
```

如果第一轮的子题 Q1 被保留，第二轮再使用 O12，那么最终 Q2 确实同时包含 O10 和 O12 带来的变化。不过目前只会把 O12 记录为本轮算子，缺少完整的算子链和贡献拆分，所以这不算受控的算子叠加。

> 注意：当前 `--num-candidates` 更接近"最多尝试多少个主/备用算子候选"，而不是"一个算子独立生成多少道题"。

## 5. 个性化算子

参考文件：`data/police_qa_testset_v2.8_场景分类汇总.md`

以"推理任务形态"为一级算子分类，以"主场景分类"为二级场景适配。

> 不建议只按照主场景分类直接设计完全独立的算子。

### 5.1 推荐路由层级

```text
进化目标/失败模式
    ↓
推理任务形态：决定使用什么算子
    ↓
主场景分类：决定算子如何场景化
    ↓
视频阶段、犯罪特征：补充约束和干扰项
    ↓
题目难度：控制进化强度
```

可以表示为：

```python
base_operator = route(进化失败模式, 推理任务形态)
scenario_adapter = 主场景分类
modifiers = 视频方法阶段 + 案件犯罪特征
strength = 题目难度 + 当前得分 + 历史进化状态
```

### 5.2 为什么推理任务形态应该优先

算子实际改变的是问题的认知要求，例如：

- **流程规范型**：增加步骤依赖、遗漏检查、异常分支。
- **比较辨析型**：增加相似项、对齐维度和适用边界。
- **判定决策型**：增加证据阈值、排除条件和结论置信度。
- **案例推理型**：增加多条线索、因果链和竞争解释。
- **原理解释型**：要求解释机制、适用条件和常见误区。
- **故障应急型**：要求定位原因、选择处置路径和补救方案。

这些属于真正可以跨场景复用的"进化动作"。而"视频身份识别""轨迹追踪""交通车辆"等主要决定题目里的对象、证据和业务约束。

### 5.3 主场景分类的正确用途

主场景分类适合做算子的场景适配器，例如：

| 基础算子 | 竞争假设与排除条件 |
| --- | --- |
| 身份识别 | 同一人、相似人员、伪装变化之间的排除 |
| 轨迹追踪 | 目标进入盲区后的多个可能出口 |
| 行为异常 | 正常行为和可疑行为之间的竞争解释 |
| 证据固定 | 事实存在、证据可用、法律证明力之间的区分 |
| 交通车辆 | 号牌、车型、损伤、轨迹和人车关系之间的交叉验证 |

这样可以保持算子机制统一，同时让生成题目符合业务场景。

### 5.4 当前分类需要进一步细化

汇总中"流程/规范型"有 243 条，占 300 条样本的 81%，如果直接作为一级分类，绝大多数题仍会进入同一个算子池。因此建议将其继续拆分为：

1. 操作步骤与先后依赖。
2. 信息完整性与遗漏检查。
3. 风险点与禁止事项。
4. 证据链构建与固定规范。
5. 异常分支与补救措施。
6. 决策节点与升级条件。
7. 多人协同与职责分工。

否则"推理任务形态"虽然方向正确，但粒度仍然不足。

### 5.5 其他分类的定位

- **原始业务场景**：标签重叠严重，例如技术侦查、情报研判、并案侦查可能同时出现，不适合做唯一算子路由。
- **视频方法/阶段**：适合视频类题目的三级路由，不能覆盖全部样本。
- **案件/犯罪特征**：适合生成干扰项、证据特征和情境变化，不适合决定基础算子。
- **难度分类**：只控制算子强度，不决定算子类型。
- **主场景分类**：适合作为业务场景池，但不应把每个场景都实现为一套完全不同的算子。

### 5.6 最终建议

不要建立完整的：

```text
主场景 × 推理形态 × 视频阶段 × 犯罪特征 × 难度
```

笛卡尔积，否则算子数量会快速膨胀，样本也不足以验证每个算子。

更合理的是三层设计：

```text
通用进化算子 O10-O18
    ↓
推理任务子类型
    ↓
场景适配 Prompt
```

例如：

```text
O12 竞争假设算子
    ↓
判定决策型
    ↓
视频身份识别适配器
    ↓
反侦查行为标签
    ↓
专业难度
```

因此，针对这份 `data/police_qa_testset_v2.8_场景分类汇总.md`，最合适的选择不是某一个分类单独承担全部路由，而是：

> **失败模式决定进化目标，推理任务形态决定基础算子，主场景决定个性化内容，其他标签作为修饰条件。**

## 6. 阶段性分步执行

分阶段推进：

### 第一阶段：单算子批量实验

```text
父题
→ 每个算子生成 K 道题
→ 每题评分 R 次
→ 比较单算子的有效率和稳定性
```

先建立可靠的单算子数据。

### 第二阶段：多层回溯

保存完整父子关系和历史分数，实现：

```text
直接父题回滚
→ 最佳有效祖先回滚
→ 原始题回滚
```

### 第三阶段：受控顺序叠加

只允许已经证明有效的算子参与组合，并限制：

- 最大链长 = 2
- 每个节点最多尝试 2 个后续算子

### 第四阶段：组合效果实验

比较：

```text
O10
O12
O10 → O12
O12 → O10
```

确认组合是否真的优于单算子，而不是仅仅增加题目长度。

## 7. 推荐搜索策略

### 先广后深

具体为：

```text
Q0
→ 广度探索多个单算子
→ 每个算子生成 K 道题并评分 R 次
→ 保留稳定有效且有差异的前 M 个节点
→ 只在这些节点上叠加第二个算子
→ 失败时回滚到最近的有效祖先
```

> 不要一开始就在所有节点上叠加所有算子。
