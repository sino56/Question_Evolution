- 建立不提交大型 experiments 的规则。

### WP1：统一评分服务

- 从 `round0_stability_probe.py` 抽取通用 trial/bundle；
- 支持 root 与候选复用；
- 缓存补齐而非全量重跑；
- 固定 reference/rubric。

### WP2：同算子 K replica

- 拆分 operator 数量与 K；
- 增加 replica ID；

- 增加 raw response 审计。

### WP3：预选与评分后选择

### WP4：Forced/Natural 实验框架

## 3. 建议验收数据集

- Tiny：3–5 条，覆盖 pass-through、硬失败、平均降分；
- Forced Pilot：每 operator 尽量 3 个 strong match；
- Natural Holdout：30–50 条未参与 Prompt 调整样本；
- Backtrack Fixtures：人工构造父/祖先/root 状态；
- Multi-boundary Fixtures：至少一个样本可命中两个不同轴；
- Pair Fixtures：至少两组 operator 顺序对照。

| 优先级 | 内容 | 原因 |
| --- | --- | --- |
| P0 | 评分包、K replica、两级选择、实验隔离 | 没有这些就无法可靠验证后续设计 |
| P1 | Prompt 版本化、forced/natural Pilot | 建立单算子证据 |

| P3 | 算子组合 | 必须在单算子有数据后进行 |

## 5. 总体完成定义

项目完成这一轮升级后，应能够对任意一个最终边界题回答：

```text
它来自哪个原始样本？
从哪个父节点生成？
使用了哪个 operator、Prompt 版本和 adapter？
同批 K 道候选是什么？
为什么 M 道进入评分？
父题和候选分别评分几次？
平均分是否下降？
GPT-5.4 是否可回答？
为什么选择它作为主链或边界？
失败时回到了哪个祖先？
该样本还发现了哪些其他能力边界？
```

若这些问题都能由结构化数据直接回答，说明系统已从“脚本串联”升级为可实验、可解释、可回溯的能力边界发现平台。
