# 00. 总体架构与阶段路线图


> 文档版本：v1.0  
> 制定日期：2026-07-11  
> 项目：Question_Evolution / PoliceQA 题目难度进化  
> 目标弱模型：Qwen3.6-27B  
> 强模型：GPT-5.4  
> 设计基线：仓库 `main` 当前主流程、`后续思路整理.md`、`样本分类画像.md` 以及本轮已确认决策。



## 1. 方案目标

本项目的核心目标不是机械增加题长、任务数量或格式约束，而是通过真实的“生成—回答—评分—反馈”闭环，在给定 PoliceQA 数据事实范围内，持续发现 Qwen3.6-27B 在不同业务场景中的可归因能力边界，并为一个原始样本保留多个有效边界题。

一期重点是建立可靠实验系统，而不是一开始就追求复杂统计最优或完整树搜索。系统需要回答四个层次的问题：

1. **算子是否有潜力**：在语义适配的样本上，某个 operator 是否能生成平均分下降的题。
2. **路由是否正确**：Profile、Diagnosis、Router、Memory 是否能自动选择合适的 operator。
3. **效果是否稳定**：降分是否在多次重新作答和评分后仍存在，而不是一次采样偶然。
4. **能力边界是否多样**：同一个样本是否能在不同能力轴上形成多个有效问题，而不是不断沿单一模板加深。

## 2. 当前主流程基线

当前推荐流水线由 `run_loop.sh` 编排，大致为：

```text
admitted_seed_samples.jsonl / data/data.jsonl
→ round0_stability_probe.py
→ profile_samples.py
→ select_evolution_candidates.py
→ operator_router.py
→ question_evolution.py
→ validate_evolved_question.py
→ light_factual_check.py
→ validate_difficulty_gain.py
→ candidate_selection.py
→ collect_answers.py
→ gen_rubric.py
→ scoring.py
→ analyze_evolution_effect.py
→ update_sample_state.py
```

已具备的能力包括：

- Round 0 多次作答评分和代表性 trial 选择；
- 样本画像与虚高诊断；
- O10–O18 operator 路由；
- 主/备 operator 的局部多候选；
- 复杂度、可回答性、事实和难度收益校验；
- 重新采集参考答案、rubric 和评分；
- 进化效果分析、回滚和 Memory；
- `score_increased` 回直接父题；
- exploration candidate 分流。

## 3. 当前关键差距

### 3.1 候选数量语义不等于 K

现有 `NUM_CANDIDATES` 更接近“最多尝试多少个主/备 operator”，通常是每个 operator 生成 1 道候选；它不是“同一 operator 独立生成 K 道”。因此必须拆分：

```text
operator_count
× generation_replicas_per_operator(K)
```

### 3.2 最终选择发生在真实评分前

当前 `candidate_selection.py` 在重新回答和评分前就选定一条主链题。结果是未选中的合格候选没有真实效果数据，无法计算一个 operator 批次的稳定性和有效候选分布。

### 3.3 节点只保留一层父快照

当前 `parent_snapshot` 只保存直接父题，不能构成完整祖先链，也不能可靠执行 `Q3 → Q2 → Q1 → Q0` 的逐级回溯。

### 3.4 有效边界会终止整个样本

当前 `effective_boundary_probe` 的状态语义更接近“样本停止”。目标方案需要改为：当前分支命中后停止该分支，但原始样本在剩余预算内继续探索其他能力轴。

### 3.5 分类画像尚未机器化

现有分类是 Markdown 中的 LLM 初步结果。运行时尚缺：

```text
reasoning_task_subtype
primary_scene
secondary_scenes
modifiers
label_source
review_status
```

而且“流程/规范型”占 243/300，粒度不足以直接驱动个性化生成。

### 3.6 Prompt 层级不完整

当前 active stack 已包含公共 operator 基础规则和 O10–O18 spec，但缺少正式的 reasoning adapter、scenario adapter 和统一版本追踪，导致同一 operator 易形成固定表面模板，场景化依赖模型临场发挥。

## 5. 总体设计原则

### 5.1 先实验，后规则

一期只把“候选平均分低于父题平均分”作为正向信号，不引入复杂最低降分阈值。其他指标先记录、分析，不立即作为硬门禁。

### 5.2 少做前置否决，多做后验真实评分

Validator 只负责明显不可回答、题外事实、严重泄漏、格式难度等硬风险；收益不确定但无致命问题的候选应在预算内进入真实压测。

### 5.3 比较必须同配置

父题和候选必须使用相同：

```text
答题模型、judge、温度、top_p、rubric、score_prompt、trial 规则
```

父题已有同配置评分包时复用缓存，不重复浪费调用。

### 5.4 节点不可变

每个 prompt 版本形成新的 `node_id`，历史节点不被覆盖。回滚是切换 `active_node_id`，而不是修改历史节点内容。

### 5.5 分支命中不等于样本结束

```text
branch_status = boundary_hit
sample_search_status = continue_other_axis / completed
```

两个状态必须分开。

### 5.6 Operator 语义稳定，Prompt 可迭代

`operator_id` 表示稳定的 failure mode 与能力轴；Prompt 文本变化通过版本字段追踪。语义实质变化不能悄悄覆盖旧 operator。

### 5.7 不引入外部事实

Reasoning/Scenario Adapter 只做结构化重述和题面事实组合，不能新增法律结论、鉴定结果、支付记录、物品性质或数据库信息。

## 6. 实施阶段

| 阶段 | 建设内容 | 主要产物 | 退出条件 |
| --- | --- | --- | --- |
| 0 | 版本化、ID、实验观测、分类抽样复核 | Prompt recipe、taxonomy v1、evaluation bundle schema | 任一候选可追溯到完整生成与评分配置 |
| 1A | 适配性约束强制验证 | forced manifest、隔离报告和 Memory | O10–O18 均获得公平验证机会或明确证据不足 |
| 1B | 自然路由验证 | router holdout 报告 | 能区分算子问题与路由问题 |
| 2 | K×M×R 和评分后选择 | preselection、candidate evaluation、final selection | M 道候选均有真实评分数据 |
| 3 | 节点化与多层回退 | evolution_nodes、backtrack controller | 能恢复父、有效祖先和 root |
| 4 | 多边界分支 | boundary_bank、fork_tasks | 一个样本可保留 1–3 个不同能力边界 |
| 5 | Adapter | taxonomy JSONL、adapter registry | 场景适配不引入题外事实，且减少固定模板 |
| 6 | 受控算子组合 | operator_sequence、顺序实验报告 | 可比较单算子与双算子增量效果 |

## 8. 明确暂缓内容

一期不做：

- 完整通用树搜索、BFS/DFS/UCB；
- 深层 parent stack 调度和任意祖先展开；
- operator 链长大于 2；
- 一次 Prompt 联合执行两个 operator；
- 全量 embedding 去重或 LLM 边界去重；
- 将 GPT-5.4 分数设为统一硬阈值；
- 将 Qwen 答题与 judge 拆成不同模型；
- 外部知识库/RAG；
- 一开始按有效率自动淘汰 operator。

## 9. 总体验收原则

系统的第一性验收不是“平均分一定下降多少”，而是以下能力均可证明：

1. 每个生成结果可追溯；
2. 父子比较同配置；
3. 同 operator 可以真正生成 K 个独立 replica；
4. M 道候选都经过真实压测；
5. 硬失败、生成失败和效果失败不混淆；
6. 强制验证与自然路由相互隔离；
7. 失败后能回到正确祖先；
8. 边界命中后可以继续寻找其他能力轴；
9. Adapter 不引入题面外事实；
10. 组合效果能拆分为单步和增量效果。
