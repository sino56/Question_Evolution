# 07. 统一数据结构与验收方案


> 文档版本：v1.0  
> 制定日期：2026-07-11  
> 项目：Question_Evolution / PoliceQA 题目难度进化  
> 目标弱模型：Qwen3.6-27B  
> 强模型：GPT-5.4  
> 设计基线：仓库 `main` 当前主流程、`后续思路整理.md`、`样本分类画像.md` 以及本轮已确认决策。



## 1. 方案目标

建立跨 Prompt、候选、评分、节点、分支、分类和实验模式统一的数据合约，使所有结果可追溯、可缓存、可回滚、可比较，并兼容现有 JSONL。

## 2. ID 设计

| ID | 作用 |
| --- | --- |
| `sample_id` | 原始样本稳定身份 |
| `root_node_id` | 原始题根节点 |
| `branch_id` | 一个能力轴探索分支 |
| `node_id` | 不可变题目版本 |
| `candidate_group_id` | 同一父节点同一 assignment 的生成组 |
| `candidate_id` | 单个 K replica |
| `evaluation_trial_id` | 一次重新作答和评分 |
| `evaluation_bundle_id` | 同一节点同配置的 trial 汇总 |
| `boundary_id` | 一个已确认能力边界 |
| `fork_task_id` | 一个待执行分支任务 |
| `experiment_id` | Forced/Natural/Pair 实验批次 |

ID 必须稳定且可重建，不能依赖数组位置。

## 4. 样本主记录

正式流水线主记录尽量保持轻量：

```json
{
  "sample_id": "6406",
  "root_node_id": "6406_root",
  "active_node_id": "6406_b01_n02",
  "best_valid_ancestor_id": "6406_b01_n01",

  "discovered_boundary_ids": ["6406_boundary_001"],
```

## 14. 报告体系

### 14.1 Prompt 报告

按 operator + prompt version：

```text
生成通过率
硬风险分布
表面模板分布
平均降分
场景覆盖
```

### 14.2 K×M×R 报告

```text
K 生成数
M 入围数
trial 完成数
父题缓存命中率
candidate positive 数
operator batch drop
```

### 14.3 回退报告

```text
回父次数
回祖先次数
回 root 次数
停止原因
循环/重复尝试检测
```

### 14.4 多边界报告

```text
每样本边界数分布
边界轴分布
重复边界数
fork 完成率
0/1/2/3 边界样本数量
```

### 14.5 Adapter 报告

```text
taxonomy 复核一致率
operator-adapter 配方分布
adapter_incompatible
题外事实风险
场景模板重复
```

### 14.6 组合报告

```text
Oa、Ob、Oa→Ob、Ob→Oa
root 总降分
第二步增量降分
顺序差异
复杂度增长
```

## 15. 兼容迁移

### 15.1 历史记录缺字段

使用 fallback：

```text
node_id = sample_id + legacy hash
root_node_id = node_id
prompt_recipe = legacy_unknown
assignment_mode = legacy_router
```

### 15.2 旧 parent_snapshot

转换为最多两层节点；无法恢复的更早历史不猜测。

### 15.3 旧 Memory

标记：

```text
prompt_version = legacy_unknown
operator_semantic_version = 1.0_assumed
```

不得与新版本强合并统计。

## 16. 测试矩阵

### 16.1 Schema

- 必填字段；
- enum；
- ID 唯一；
- parent 存在；
- root 一致；
- chain depth 不超过配置。

### 16.2 Prompt

- recipe hash；
- semantic version；
- adapter 装配；
- source fact map。

### 16.3 K×M×R

- 同 operator K=3；
- M=2；
- R=1/3/5 状态；
- 父题同配置缓存；
- trial 去重。

### 16.4 回退

- 父、祖先、root；
- 硬/软失败计数；
- manual review 不回退；
- 无循环。

### 16.5 多边界

- branch/sample 状态分离；
- 上限 3；
- 新轴 root；
- 同轴 ancestor；
- 重复不计数。

### 16.6 实验隔离

- forced 不写 production；
- natural 不读取 forced override；
- report 可对照。

## 17. 分阶段验收

### 阶段 0：数据与观测

- 所有候选可追溯到 recipe、assignment 和配置；
- 50 条 taxonomy 复核完成；
- schema 校验可在 CI 运行。

### 阶段 1：Prompt + Forced/Natural

- 两种模式可独立运行；
- O10–O18 有公平验证或证据不足状态；
- Memory 隔离。

### 阶段 2：K×M×R

- 每 operator K=3；
- M 候选均有 R=3；
- 评分后选择；
- 父题评分包可复用。

### 阶段 3：回退

- 完整祖先链；
- 默认规则准确；
- 不丢评分与 rubric。

### 阶段 4：多边界

- 一个样本可保留多个边界；
- 最多 3；
- 不强制凑满。

### 阶段 5：Adapter

- 不引入题外事实；
- taxonomy 和 adapter 配方可审计；
- 跨场景 operator 语义稳定。

### 阶段 6：组合

- 最大链长 2；
- 顺序和增量可比较；
- 失败可回第一步。

## 18. 端到端验收样例

选择一个样本：

1. Root 完成 R=3 评分；
2. Forced O11 生成 K=3；
3. 预选 M=2；
4. 两道均 R=3；
5. 选出平均分下降题，GPT-5.4 可回答；
6. 创建 verified node 和 boundary；
7. 样本仍有 O17 新轴，创建 root fork；
8. O17 失败两次，正确回 root 或有效祖先；
9. 最终样本保留 1–2 个边界；
10. 全部数据可由 ID 串联，断点重跑无重复记录。

## 19. 发布门槛

正式合并主流程前必须：

- `pytest -q` 全部通过；
- tiny pipeline 端到端通过；
- schema validation 通过；
- 断点续跑测试通过；
- Forced/Natural Memory 不交叉；
- 历史输入兼容测试通过；
- 不提交 API key、实验大文件和私有模型输出。
