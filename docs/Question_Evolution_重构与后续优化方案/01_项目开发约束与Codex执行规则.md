# 01. 项目开发约束与 Codex 执行规则

## 1. 使用方式

本文件可作为：

- 仓库根目录 `AGENTS.md` 的核心补充；
- 每次交给 Codex 的任务前置约束；
- 方案评审和代码审查的最高优先级规则；
- 防止后续 vibe coding 再次产生规则膨胀的开发宪法。

## 2. 项目最高目标

本项目是模型能力边界发现系统，不是候选题拒绝系统。

主要目标是：

```text
通过题目进化，使 Qwen 在进化题上的真实平均得分相对于父题下降，
并根据具体失分内容分析其能力边界。
```

主要效果指标：

```text
score_drop = parent_score_mean - candidate_score_mean
```

所有算子、Prompt、Router、Memory、树搜索和辅助分析，最终都必须通过真实 score drop 评价。

## 3. 禁止目标漂移

不得将以下指标作为项目主要成功标准：

- validator 通过率；
- 题目格式规范率；
- Router 命中率；
- GPT 预测难度收益；
- Memory 收敛速度；
- 所有候选一次生成成功；
- 全流程无失败；
- 所有样本最终都找到边界；
- 所有边界都满足正式数据集质量要求。

这些只能作为辅助指标。

## 4. 核心闭环不可破坏

项目必须始终保留一条不依赖复杂路由、Memory、Adapter、树搜索和组合机制的最小闭环：

```text
父题 Qwen 评分
→ GPT 分析与算子选择
→ 候选题生成
→ 致命错误检查
→ 参考答案生成
→ rubric 与 score prompt 生成
→ Qwen 作答与评分
→ 前后效果分析
```

任何新机制都不得使该闭环无法独立运行。

## 5. 前置硬拒绝范围

只有以下情况可以在首次真实评分前阻止候选：

1. 候选题为空；
2. 候选题与父题完全相同；
3. 样本 ID、父节点 ID 或关键数据关联丢失；
4. JSON 或结构化输出无法解析；
5. 模型调用失败且有限重试后仍无结果；
6. 无法生成参考答案；
7. rubric 为空或无法用于评分；
8. 题目存在明确内部逻辑矛盾，无法形成任何确定评价；
9. 数据结构损坏，继续执行会污染其他样本。

以下情况默认只能作为 soft flag：

- 多主轴；
- 多任务；
- 题目较长；
- 包含多个条件；
- 反事实数量较多；
- 格式复杂；
- 可能依赖背景知识；
- 与历史题型相似；
- GPT 预测难度收益不足；
- 样本与 operator 仅 possible match；
- source fact map 不完整；
- Adapter 存在潜在冲突。

## 6. 实验失败不是系统错误

必须区分：

### 6.1 系统错误

- API 失败；
- 文件损坏；
- schema 错误；
- ID 冲突；
- 配置不一致；
- 输出无法解析；
- trial 错配；
- 缓存污染。

这些可以触发重试、隔离或阶段停止。

### 6.2 实验失败

- 没有降分；
- 得分升高；
- Router 选错；
- operator 无效；
- 第二个 operator 没有增量；
- 候选重复；
- Qwen 得分波动大。

这些必须保留记录，不得默认触发系统崩溃或全局规则收紧。

## 7. 默认禁止的修改

除非任务明确授权且已有实验数据支持，否则禁止：

- 新增 validator；
- 新增全局硬阈值；
- 新增状态枚举；
- 新增 Memory Bank；
- 新增全局 operator blacklist；
- 新增多层 fallback 链；
- 新增一个模型调用阶段；
- 同时修改进化 Prompt、rubric Prompt 和 scoring 逻辑；
- 根据一次失败永久降低 operator 权重；
- 将 possible match 全部排除；
- 将 validator 预测结果设为主效果指标；
- 将 tree search、Adapter 和 operator combination 一次性同时接入。

## 8. 优先允许的修改

优先采用：

- 只修改一个 operator Prompt；
- 增加 Prompt 版本和配置 hash；
- 增加旁路观察字段；
- 将硬门禁改为软标签；
- 增加真实评分覆盖率统计；
- 增加对照实验脚本；
- 删除重复规则；
- 合并状态枚举；
- 增加 feature flag；
- 增加不可变节点和缓存；
- 增加候选级失败隔离；
- 增加架构不变量测试。

## 9. 每次开发前必须输出

Codex 在修改代码前必须先给出：

1. 本轮实验观察到的客观事实；
2. 有数据支持的结论；
3. 未被验证的假设；
4. 是否真的需要修改代码；
5. 是否可以只修改 Prompt；
6. 最小可验证改动；
7. baseline 与对照组；
8. 成功判据；
9. 失败与删除判据；
10. 回滚方法；
11. 新增了哪些复杂度；
12. 本次明确不做什么。

没有完成以上分析，不得直接实施大规模方案。

## 10. 每项新规则必须附带删除条件

新增任何永久规则时，必须写明：

```text
触发该规则的数据证据：
预期节省的成本：
可能误拒绝的候选：
A/B 对照方式：
观察周期：
删除条件：
回滚开关：
```

没有删除条件的规则不得进入主分支。

## 11. 搜索和回溯约束

- 搜索控制器只能选择下一次展开节点；
- 回溯不能覆盖或删除历史节点；
- 单次失败只能影响当前 parent + operator + Prompt version + surface form；
- 单样本失败不得形成全局 operator blacklist；
- 树搜索必须可关闭；
- tree mode 关闭后，core mode 必须仍能运行。

## 12. 算子叠加约束

- 禁止一个 Prompt 同时要求执行两个 operator；
- 只允许顺序改写；
- 第一阶段最大链长为 2；
- 必须分别记录第一步效果、第二步增量和总效果；
- 第二步没有增量时不得判定组合有效；
- 组合实验不得直接污染 Natural Router 的 Memory。

## 13. 复杂度预算

每轮开发默认遵循：

```text
新增一个永久机制
→ 至少删除、合并或降级一个旧机制
```

代码审查必须报告：

- 新增文件数；
- 新增状态数；
- 新增配置项数；
- 新增模型调用数；
- 新增硬门禁数；
- 删除或降级的规则数。

## 14. 最终原则

```text
少加规则，多做实验。
少做前置否决，多做后验反馈。
少惩罚整个 operator，多记录具体失败形态。
少追求静态完美，多追求真实 score drop。
```
