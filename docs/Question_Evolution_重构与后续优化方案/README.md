# Question_Evolution 重构与后续优化方案文档包

## 1. 文档包目的

本方案包用于指导 `Question_Evolution` 项目从当前“规则逐渐膨胀、前置校验阻碍候选进入真实评分”的状态，调整为一套以真实 Qwen 评分为核心证据、支持轻量树搜索、局部回溯与受控算子叠加的稳定实验系统。

项目的核心目标保持不变：

> 对原始题目进行定向进化，使目标弱模型 Qwen 在进化题上的真实得分下降，并根据具体失分内容归因分析其能力边界。

本方案不追求一次性重写全部代码，也不要求立即实现完整树搜索、复杂多 Agent 调度或自动算子淘汰。后续开发必须先恢复稳定、可解释、可对照的最小实验闭环，再逐步增加搜索能力。

## 2. 核心调整

本方案对原有思路做出以下关键调整：

1. 将 Qwen 真实评分重新设为主要效果证据。
2. 将复杂度、事实风险、格式风险、题外知识等校验默认降级为旁路观察。
3. 区分能力边界探索系统与最终数据质量整理系统。
4. 将 K×M×R 调整为“先让全部非致命候选获得首次真实评分，再决定哪些候选追加评分”。
5. 树搜索只负责选择从哪个节点继续探索，不负责修改历史结果。
6. 回溯只改变下一次展开节点，不覆盖或删除历史节点。
7. 算子叠加采用顺序改写，不采用一个 Prompt 同时执行多个算子。
8. 所有复杂机制必须可关闭、可回滚，并通过对照实验验证净收益。

## 5. 最高原则

```text
少加规则，多做真实实验。
少做前置否决，多做后验分析。
少消除失败，多利用失败信息。
少追求静态流程完美，多追求真实 score drop。
```

任何新模块都必须回答：

- 它解决了什么已被实验数据证实的问题？
- 为什么不能通过 Prompt 调整或后验分析解决？
- 是否会减少候选进入真实 Qwen 评分的机会？
- 如何进行 A/B 对照？
- 什么结果出现时应删除或回滚该机制？
