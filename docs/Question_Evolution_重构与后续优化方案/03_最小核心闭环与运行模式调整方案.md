# 03. 最小核心闭环与运行模式调整方案

## 1. 调整目标

当前首要任务不是继续增加搜索机制，而是建立一条长期稳定、可复现、可作为所有复杂功能 baseline 的最小核心闭环。

## 2. 最小核心闭环

推荐流程：

```text
Step 0 读取已准入种子样本
Step 1 Qwen 对父题进行稳定评分
Step 2 GPT 分析题目、回答、rubric 和得分
Step 3 选择 operator
Step 4 生成 K 个候选题
Step 5 仅执行致命错误检查
Step 6 为候选生成参考答案
Step 7 生成 rubric 和 score prompt
Step 8 Qwen 对候选作答与评分
Step 9 计算 score drop
Step 10 后验分析并保存所有结果
```

第一阶段暂不要求：

- 多层祖先回退；
- fork task；
- Adapter registry；
- operator pair；
- 自动 operator 淘汰；
- 全局 Memory 收敛；
- 复杂难度收益预测门禁。

## 3. 三种兼容运行模式

建议增加统一参数：

```text
PIPELINE_MODE=core|shadow|strict
```

### 3.1 core

执行：

```text
generation
→ fatal_check
→ answer
→ rubric
→ scoring
→ effect_analysis
```

用途：

- 验证主流程；
- 调试模型接口；
- 作为稳定 baseline；
- 复现最原始能力边界实验。

### 3.2 shadow

执行：

```text
core
+ complexity observation
+ factual observation
+ answerability observation
+ difficulty prediction
+ duplicate analysis
```

但所有观察结果只能写标签。

约束：

```text
shadow 模式与 core 模式的非致命候选首次评分数必须相同
```

### 3.3 strict

保留旧流程行为，用于：

- 旧实验复现；
- 后验正式数据整理；
- 比较 validator 拦截效果；
- 评估 strict 的 false reject。

strict 不应继续作为能力边界发现默认模式。

## 4. Fatal Check 设计

建议新增或重构统一致命检查：

```text
fatal_check.py
```

只判断：

```text
empty_prompt
same_as_parent
missing_sample_id
missing_parent_id
unparseable_output
irrecoverable_model_failure
reference_generation_failure
rubric_generation_failure
invalid_scoring_contract
internal_logical_contradiction
```

输出：

```json
{
  "fatal": false,
  "fatal_type": null,
  "reason": "",
  "retryable": false
}
```

其他风险进入 `observations`。

## 5. 候选级失败隔离

当前流水线不应因为一条候选失败而中止同组全部候选。

推荐语义：

```text
单候选 API 失败
→ 有限重试
→ 标记 candidate_failed
→ 继续其他候选
```

只有以下阶段级错误才停止本批次：

- 输入文件损坏；
- ID 关联错误；
- 父节点不存在；
- 大规模输出数量错位；
- 配置 hash 冲突；
- trial 被错误关联；
- 正式输出无法保证原子性。

## 6. 配置一致性

父题与候选必须使用相同：

```text
answer_model
judge_model
temperature
top_p
rubric
score_prompt
trial_rule
scoring_code_version
```

如果配置不一致：

```text
bundle_status = invalid_config_mismatch
```

不得计算正式 score drop。

## 7. 基础结果结构

每个候选至少保存：

```json
{
  "sample_id": "...",
  "candidate_id": "...",
  "parent_node_id": "...",
  "operator_id": "...",
  "prompt_version": "...",
  "prompt": "...",
  "fatal_check": {},
  "observations": {},
  "evaluation_bundle_id": "...",
  "score_drop": 0.12,
  "effect_status": "positive_signal"
}
```

## 8. 旧流程兼容

现有严格模块不必立即删除。

可以通过编排层控制：

```text
core:
  skip validate_evolved_question
  skip light_factual_check as gate
  skip validate_difficulty_gain as gate
  skip candidate_selection before scoring

shadow:
  run validators
  attach results
  continue all nonfatal candidates

strict:
  preserve legacy gating
```

## 9. 对照实验

使用同一批样本、同一模型和同一随机配置比较：

```text
core
shadow
strict
```

重点指标：

```text
generated_candidate_count
first_score_candidate_count
confirmed_score_candidate_count
average_score_drop
positive_candidate_count
fatal_failure_count
strict_would_reject_count
strict_false_reject_count
cost_per_boundary
```

## 10. 验收标准

1. core 模式能独立跑通完整闭环；
2. shadow 模式不减少首次评分候选；
3. strict 模式保留旧行为；
4. 单候选失败不会终止整个 operator 批次；
5. 每个候选可追溯到 Prompt、operator 和配置；
6. 父题与候选配置不一致时不会错误计算 score drop；
7. 所有结果可断点续跑；
8. 关闭所有增强开关后仍可复现 core baseline。
