# 方案二：轻量增强与记忆收敛实现方案

## 1. 方案定位

本方案是 Question_Evolution 项目的第二批增强方案，应在“方案一：低风险核心闭环”已经完成并跑过小样本实验后再实施。

方案一解决的是：

```text
候选能否进入真实评分闭环
```

方案二解决的是：

```text
在不显著增加复杂度的前提下，如何让系统逐步利用失败经验收敛搜索空间
```

本方案仍然坚持轻量原则：

1. 不把所有风险都提前变成 hard reject；
2. 不默认引入复杂 LLM fact validator；
3. 不修改 rubric 生成和 scoring；
4. failure memory 慢收敛；
5. template/rubric 风险主要用于排序和报告；
6. 只有连续同类失败后才影响路由。

---

## 2. 实施前提

实施方案二前，应确认方案一已经满足：

1. `candidate_selection.py` 支持三分流；
2. `exploration_candidate` 能进入真实评分；
3. `score_increased_after_evolution` 不再被当成成功终止态；
4. O1/O2/O4/O8 显式脚手架减少；
5. 至少有候选进入真实评分闭环；
6. 已经能观察以下指标：
   - `main_chain_selection_rate`
   - `exploration_selection_rate`
   - `any_scoring_entry_rate`
   - score drop / score increase

如果方案一尚未完成，不建议提前实施本方案。

---

## 3. 本方案主要做什么

方案二包含五个轻量增强：

1. `light_factual_check`
2. template/rubric soft penalty
3. failure memory 最近 N 轮窗口
4. operator_router 轻量 downrank / avoid
5. `uncertain_low_probe` 可选开关

这些机制都应保持“轻量”和“慢收敛”，不能重新变成强治理系统。

---

## 4. 本方案不做什么

本阶段仍暂缓：

1. 不新增复杂 LLM fact validator 默认流程；
2. 不新增 `validate_rubric_specificity.py`；
3. 不做 Generic Answer Rubric Probe；
4. 不修改 `gen_rubric.py`；
5. 不修改 rubric prompt；
6. 不修改 score prompt；
7. 不修改 scoring.py；
8. 不把 template/rubric 风险单独作为 hard reject；
9. 不让第一次失败就强惩罚 operator；
10. 不对整个 operator 做粗暴封禁。

---

## 5. light_factual_check

### 5.1 位置

放在：

```text
validate_evolved_question.py
→ light_factual_check
→ validate_difficulty_gain.py
```

原因：

1. 它是低成本规则检查；
2. 可以先拦掉明显事实错误；
3. 可以减少后续 LLM 验证成本；
4. 高置信事实错误没有必要进入 difficulty gain 判断。

### 5.2 第一版只查高置信事实错误

hard reject 只包括：

1. 数字、时间、金额、距离、比例新增或冲突；
2. 原题明确实体名被替换；
3. 候选题出现原题没有的确定性证据结果；
4. 候选题明确修改原题核心数值；
5. 候选题明确修改原题核心实体。

### 5.3 只 warning 的情况

以下情况先不 hard reject：

1. 一般场景收窄；
2. 处置边界限制；
3. 可用材料限制；
4. 结论措辞变化；
5. 背景信息轻微改写；
6. 删除关键限制。

其中“删除关键限制”通常是语义判断，第一版不要靠轻量规则硬拒绝。只有出现明确文本级反转时才 hard reject，例如：

```text
原题：不得补充新材料
候选题：可以补充新材料
```

否则只作为 warning。

### 5.4 输出字段

```json
{
  "light_factual_check": {
    "passed": true,
    "fatal_errors": [],
    "warnings": [],
    "risk_tags": []
  }
}
```

### 5.5 candidate_selection 使用方式

```text
fatal_errors 非空 → hard_reject
warnings 非空 → selection_score - 0.05
```

不要将 warning 直接作为 hard reject。

## 6. template_affordance_risk soft penalty

### 6.1 原则

`template_affordance_risk` 只作为排序信号，不作为单独 hard reject。

### 6.2 使用方式

默认扣分：

```text
template_affordance_risk = medium → -0.05
template_affordance_risk = high → -0.10
```

### 6.3 hard reject 例外

只有同时满足以下条件时，才 hard reject：

```text
generic_answer_success_likelihood >= 0.85
and specific_fact_dependency_score <= 0.35
```

如果没有这两个字段，则不能仅凭主观模板判断 hard reject。

### 6.4 推荐字段

```json
{
  "template_affordance_risk": "low / medium / high",
  "surface_form_family": "generic_cautious_judgment / evidence_relation_comparison / counterfactual_boundary / fact_binding_review / unknown",
  "template_risk_reason": "简要说明风险"
}
```

---

## 7. rubric_shortcut_risk 报告化

### 7.1 当前阶段不作为门禁

`rubric_shortcut_risk` 只进入报告，不参与 hard reject。

### 7.2 不修改 rubric 相关模块

本方案仍不修改：

1. `gen_rubric.py`
2. rubric prompt
3. score_prompt
4. scoring.py

### 7.3 使用目的

仅用于观察：

```text
候选题是否容易诱导后续 rubric 奖励套话
```

如果该风险频繁出现，后续可以再进入 rubric 质量阶段，但不在当前阶段解决。

---

## 8. failure memory 最近 N 轮窗口

### 8.1 为什么需要窗口

如果 memory 永久生效，早期失败会长期污染路由，导致搜索空间过早收缩。

因此需要最近 N 轮窗口。

### 8.2 默认窗口

```text
FAILURE_MEMORY_WINDOW_ROUNDS = 3
```

可配置为：

```text
3 或 5
```

### 8.3 同类失败定义

同类失败必须同时匹配：

```text
sample_signature
+ operator_used
+ surface_form_family
+ failure_type
```

其中字段兼容：

| 语义 | 主字段 | 兼容别名 |
|---|---|---|
| operator | operator_used | operator_id, candidate_operator |
| sample signature | sample_signature | sample_profile_signature |
| score before | score_rate_before | score_before |
| score after | score_rate_after | score_after |
| score delta | delta_score_rate | score_delta |
| surface form | surface_form_family | question_surface_form |
| failure type | failure_type | effect_label |

### 8.4 慢收敛规则

```text
最近 N 轮内同类失败 1 次 → warn_only
最近 N 轮内同类失败 2 次 → downrank operator + surface form
最近 N 轮内同类失败 3 次 → avoid operator + surface form 或切 operator family
```

注意：惩罚对象是：

```text
operator_used + surface_form_family
```

不是整个 operator。

## 9. operator_router 轻量消费 memory

### 9.1 warn_only

当同类失败只有 1 次时，只输出警告：

```json
{
  "operator_route": {
    "memory_warnings": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "failure_count": 1,
        "action": "warn_only"
      }
    ]
  }
}
```

### 9.2 downrank

当同类失败达到 2 次：

```json
{
  "operator_route": {
    "downrank_operator_surface_forms": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "reason": "repeated_negative_gain"
      }
    ]
  }
}
```

### 9.3 avoid

当同类失败达到 3 次：

```json
{
  "operator_route": {
    "avoid_operator_surface_forms": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "reason": "repeated_negative_gain"
      }
    ]
  }
}
```

### 9.4 不过度惩罚

不要因为某个 surface form 失败就完全禁用整个 operator。

---

## 10. uncertain_low_probe

### 10.1 定位

`uncertain_low_probe` 是探索开关，不是默认主流程。

### 10.2 默认关闭

```text
ENABLE_UNCERTAIN_LOW_PROBE = false
```

### 10.3 开启条件

当开关开启，且样本满足：

```text
admission_status = uncertain_low
and overscore_diagnosis.is_worth_evolving = true
and stable_score >= UNCERTAIN_LOW_PROBE_MIN_SCORE
```

则允许低预算探索。

默认：

```text
UNCERTAIN_LOW_PROBE_MIN_SCORE = 0.55
```

### 10.4 预算

```text
recommended_evolution_budget = 1
```

### 10.5 目的

扩大探索面，避免系统长期只进化少数 stable_high / borderline 样本。

## 12. 第二批测试要求

至少新增测试：

1. `light_factual_check.fatal_errors` 非空时 hard reject；
2. `light_factual_check.warnings` 非空时只扣分，不 hard reject；
3. template medium 扣 0.05；
4. template high 扣 0.10；
5. template high 但没有量化字段时不 hard reject；
6. generic answer success >= 0.85 且 fact dependency <= 0.35 时 hard reject；
7. rubric_shortcut_risk 只写报告，不参与 hard reject；
8. 最近 N 轮内同类失败 1 次 → warn_only；
9. 最近 N 轮内同类失败 2 次 → downrank；
10. 最近 N 轮内同类失败 3 次 → avoid；
11. operator_router 不因单个 surface form 失败而禁用整个 operator；
12. `ENABLE_UNCERTAIN_LOW_PROBE=false` 时 uncertain_low 仍 stop；
13. `ENABLE_UNCERTAIN_LOW_PROBE=true` 且满足条件时，budget=1。

---

## 13. 实施顺序

### 第一步：实现 light_factual_check

先只做高置信规则。

### 第二步：接入 candidate_selection soft penalty

将 warning / template risk 纳入 selection score。

### 第三步：实现 failure memory N 轮窗口

先只统计，不改变 router。

### 第四步：operator_router 消费 memory

按 warn_only → downrank → avoid 分层接入。

### 第五步：实现 uncertain_low_probe 开关

默认关闭，只在实验时打开。

### 第六步：小样本实验

观察通过率、score drop、memory 使用情况，再决定是否进一步增强。

---

## 14. 第二批验收标准

完成后应满足：

1. light factual check 只拦高置信 fatal error；
2. factual warning 不会 hard reject；
3. template risk 和 rubric risk 不会单独压死候选；
4. failure memory 有最近 N 轮窗口；
5. 同类失败 1 次只记录；
6. 同类失败 2 次才降权；
7. 同类失败 3 次才 avoid；
8. router 不会过度惩罚整个 operator；
9. uncertain_low_probe 可配置开启；
10. 整体 candidate_selection 通过率没有明显崩塌；
11. 系统复杂度仍低于强治理版本。
