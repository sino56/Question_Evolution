# 方案三：低风险核心闭环增量实现方案

## 1. Context｜任务背景

### 1.1 当前项目状态

当前 `Question_Evolution` 项目已经不是空白实现，代码中已经形成了阶段化流水线：

```text
round0_stability_probe.py
→ profile_samples.py
→ select_evolution_candidates.py
→ operator_router.py
→ question_evolution.py
→ validate_evolved_question.py
→ validate_difficulty_gain.py
→ candidate_selection.py
→ collect_answers.py
→ gen_rubric.py
→ scoring.py
→ analyze_evolution_effect.py
→ update_sample_state.py
```

因此，本方案不是“重做已有功能”，而是在现有阶段 1-6 的基础上做增量开发。核心目标是补上一个受控探索通道，让已有难度收益验证不再把所有非强通过候选都挡在真实评分之前。

### 1.2 代码审视结论

基于当前代码，已有能力和缺口如下：

| 模块 | 当前已有能力 | 当前缺口 | 本方案处理方式 |
|---|---|---|---|
| `validate_difficulty_gain.py` | 已生成 `difficulty_gain_validation`，支持 `clear_gain`、`probable_gain`、`weak_gain`、`no_gain`、`leakage_or_simplification`、`format_complexity_only`、`axis_shift`、`unanswerable_or_external`、`needs_manual_review`、`not_applicable` | `weak_gain`、`no_gain`、`needs_manual_review` 默认无法进入真实评分 | 不重写 validator，优先在 `candidate_selection.py` 增量解释这些标签 |
| `candidate_selection.py` | 已按 candidate group 选择候选；当前 `PASS_DIFFICULTY_GAIN_LABELS = {clear_gain, probable_gain, not_applicable}` | 非 pass label 基本都会被拒绝，缺少 exploration 分流 | 增加 `candidate_flow` 与 exploration 预算逻辑，保留已有 selected/status 字段 |
| `schemas/*.schema.json` | 多数 schema `additionalProperties = true` | 新字段可以兼容加入，但 `difficulty_gain_label` enum 不应随意扩展 | 第一批不新增 validator label；如需兼容 `borderline_gain/uncertain_gain`，只在 selection 层做 alias |
| `analyze_evolution_effect.py` | 已识别 `score_increased` | 未显式输出 `score_increased_after_evolution` | 保留 `effect_label = score_increased`，增量增加负收益布尔标记 |
| `update_sample_state.py` | 已写入 state、operator memory、failure memory | 当前 `score_increased` 会被映射到 `validated_high_score_sample`，语义与负收益冲突 | 做最小语义修正：score increase 不进入 terminal success，只写 failure memory |
| O1/O2/O4/O8 prompt | 已有 operator spec | 文本存在显式 A/B、层级、排序、双门槛脚手架 | 只替换现有字段文本，不扩展 `OperatorPromptSpec` |
| `run_loop.sh` | 已接入 `validate_difficulty_gain.py` 和 `candidate_selection.py` | 暂无 exploration 预算参数 | 第一批尽量不改主流程顺序，只在必要时向 `candidate_selection.py` 追加可选参数 |

### 1.3 本方案目标

在不破坏现有流水线、schema 和测试语义的前提下，新增低风险探索能力：

```text
已有强通过候选 → 继续进入主链
已有致命失败候选 → 继续 hard reject
已有弱收益/不确定候选 → 在预算内进入 exploration
真实评分结果 → 决定是否有效，而不是仅由静态 validator 决定
```

## 2. Request｜具体任务

请在现有代码基础上进行增量实现，而不是重写已有功能。

具体包括：

1. 审视并保留现有 pipeline、字段、状态和测试约定；
2. 在 `candidate_selection.py` 增量增加候选流分类；
3. 允许部分非强通过候选以 `exploration_candidate` 进入真实评分；
4. 为 exploration 增加每组和每轮预算上限；
5. 保留 `effect_label = score_increased`，同时新增 `score_increased_after_evolution` 负收益标记；
6. 最小修正 `update_sample_state.py` 中 score increase 的后续状态，不再把它转成 terminal success；
7. 对 O1/O2/O4/O8 做现有字段内的文本替换；
8. 增加或调整测试，确保旧功能不回退。

## 3. 增量开发原则

### 3.1 不重写已有阶段

不要重排 `run_loop.sh` 的主流程。当前 `validate_difficulty_gain.py → candidate_selection.py → collect_answers.py → gen_rubric.py → scoring.py` 的链路已经存在，本方案只在现有节点中补充分流逻辑。

### 3.2 不删除旧字段

必须保留现有输出字段，例如：

```text
candidate_selection.selected
candidate_selection.selected_candidate_id
candidate_selection.selected_operator
candidate_selection.selection_status
candidate_selection.selection_score
candidate_selection.selection_reason
candidate_selection.rejected_candidates
effect_analysis.effect_label
evolution_state.stop_status
```

新增字段只能作为补充，不应替代旧字段。

### 3.3 不强制迁移旧数据

历史 JSONL 中可能没有 exploration 字段。实现时应兼容缺失字段，不能要求重跑所有历史实验。

### 3.4 不扩展 validator label enum

当前 `schemas/difficulty_gain_validation.schema.json` 中没有 `borderline_gain` 和 `uncertain_gain`。第一批不建议修改该 enum。

如果输入数据中未来出现这些 label，可以在 `candidate_selection.py` 内做兼容 alias：

```text
borderline_gain → weak_gain 类 exploration 信号
uncertain_gain → needs_manual_review 类 exploration 信号
```

但不要要求 `validate_difficulty_gain.py` 第一批产出新 label。

## 5. 候选分流增量设计

### 5.1 新增字段

在 `candidate_selection` 输出中增量增加：

```json
{
  "candidate_flow": "main_chain_candidate | exploration_candidate | hard_reject | pass_through_candidate",
  "selected_for_exploration": false
}
```

要求：

1. 旧字段继续存在；
2. 旧测试依赖的 `selection_status` 尽量保持；
3. 只有 exploration 选中进入后续评分时，`selected_for_exploration = true`。

### 5.2 main_chain_candidate

保持现有强通过逻辑。

条件：

```text
validation_result.passed = true
and difficulty_gain_validation.passed = true
and difficulty_gain_label in {clear_gain, probable_gain, not_applicable}
```

输出建议：

```json
{
  "candidate_flow": "main_chain_candidate",
  "selected_for_exploration": false,
  "selection_status": "selected_after_difficulty_gain_validation"
}
```

### 5.3 hard_reject

保持对致命坏题的拒绝。

触发条件：

1. `validation_result.passed = false`；
2. `difficulty_gain_label` 属于：
   - `leakage_or_simplification`
   - `format_complexity_only`
   - `axis_shift`
   - `unanswerable_or_external`
3. `risk_tags` 命中 hard risk；
4. 必要字段缺失或 JSON 解析失败。

当前代码中应纳入 hard risk 的标签至少包括：

```text
missing_premise_named
conclusion_hint_revealed
answer_path_scaffolded
external_fact_dependency
format_difficulty_only
axis_shift
```

可兼容新增或历史标签：

```text
unanswerable_or_external
multi_axis
direct_answer_leakage
```

输出建议：

```json
{
  "candidate_flow": "hard_reject",
  "selected_for_exploration": false,
  "selection_status": "hard_reject"
}
```

### 5.4 exploration_candidate

这是本方案的核心增量。

候选没有致命问题，但静态收益不够强时，可进入受控 exploration。

当前代码优先支持以下现有 label：

```text
weak_gain
needs_manual_review
no_gain
```

其中：

1. `weak_gain`：默认可作为 exploration 候选；
2. `needs_manual_review`：若没有 hard risk，可作为 exploration 候选；
3. `no_gain`：不能默认进入 exploration，必须满足“无 hard risk 且存在探索价值”。

兼容未来或外部输入 alias：

```text
borderline_gain → exploration_candidate
uncertain_gain → exploration_candidate
```

输出建议：

```json
{
  "candidate_flow": "exploration_candidate",
  "selected_for_exploration": true,
  "selection_status": "selected_for_exploration",
  "selection_reason": "候选无致命问题但静态收益不确定，进入低预算真实评分验证。"
}
```

### 5.5 pass_through_candidate

候选无致命风险，但没有足够探索价值时，不进入真实评分，也不写负面 memory。

典型条件：

```text
no_gain
and no hard risk
and no novelty
and no target_failure_pressure
and no exploration value
```

输出建议：

```json
{
  "candidate_flow": "pass_through_candidate",
  "selected_for_exploration": false,
  "selection_status": "not_selected_no_exploration_value"
}
```

## 6. no_gain 增量分流规则

`no_gain` 不再自动等于 hard reject，但也不能自动进入 exploration。

### 6.1 no_gain → exploration_candidate

满足以下条件时才可探索：

```text
no_gain
and no hard risk
and exploration value exists
```

`exploration value exists` 可由以下信号之一支持：

1. `recommended_action` 是 `needs_manual_review`、`admit_low_priority` 或类似低优先级保留信号；
2. `risk_tags` 只包含 soft risk；
3. `difficulty_gain_score` 接近 `BORDERLINE_DIFFICULTY_GAIN_SCORE`；
4. `weak_probe.target_failure_match = true`；
5. 候选引入了不同 surface form，且仍保持原能力轴。

### 6.2 no_gain → hard_reject

满足以下任一条件时仍 hard reject：

```text
no_gain
and risk_tags contains hard risk
```

或：

```text
no_gain
and validation_result.passed = false
```

或：

```text
no_gain
and generic answer 显然可以高分
```

### 6.3 no_gain → pass_through_candidate

如果没有 hard risk，但也没有 exploration value，则只作为普通未采纳候选处理。

要求：

1. 不进入真实评分；
2. 不写 failure memory；
3. 不惩罚 operator；
4. 可记录在 `rejected_candidates` 中供审计。

---

## 7. exploration 预算边界

### 7.1 每组上限

```text
MAX_EXPLORATION_PER_GROUP = 1
```

同一个 `candidate_group_id` / `sample_id` / `index` 下，最多选择 1 个 exploration candidate。

### 7.2 每轮总上限

```text
MAX_EXPLORATION_CANDIDATES_PER_ROUND = 5
```

建议实现为 `candidate_selection.py` 的可选 CLI 参数或环境变量，例如：

```text
--max-exploration-candidates-per-round 5
```

不传参数时使用默认值，保证旧调用方式仍可运行。

### 7.3 排序规则

同一组内优先级：

```text
main_chain_candidate
→ exploration_candidate
→ pass_through_candidate
→ hard_reject
```

当存在 main chain candidate 时，继续按现有 `selection_score` 选择最高者。  
当没有 main chain candidate 但存在 exploration candidate 时，选择最高分 exploration 候选补位。

## 8. selection_score 增量调整

当前 `score_candidate()` 已使用：

```text
difficulty_gain_score
no_leakage_score
competitive_judgment_score
weak_probe_score
validation_quality_score
```

第一批不要重写该公式。只需要在 exploration 场景中增加 cap 或轻量扣分：

```text
weak_gain → selection_score cap at 0.60
needs_manual_review → selection_score cap at 0.58
no_gain with exploration value → selection_score cap at 0.55
hard risk → hard_reject
```

如果实现成本更低，也可以第一批只沿用现有分数排序，新增测试保证 exploration 不超过预算。

## 9. template_simplification 边界

当前代码没有稳定字段：

```text
generic_answer_success_likelihood
specific_fact_dependency_score
```

因此第一批不应强依赖这两个字段。

增量处理方式：

1. 如果字段存在，且同时满足以下条件，可 hard reject：

```text
generic_answer_success_likelihood >= 0.85
and specific_fact_dependency_score <= 0.35
```

2. 如果字段不存在，不因为缺字段而 hard reject；
3. 模板化风险仅作为 soft penalty 或 exploration 降权；
4. 不新增复杂模板风险系统。

## 10. score_increased_after_evolution 增量处理

### 10.1 保留现有 effect_label

`analyze_evolution_effect.py` 当前已经输出：

```text
effect_label = "score_increased"
```

第一批不重命名该 label。

### 10.2 新增负收益标记

在 `effect_analysis` 中增量增加：

```json
{
  "score_increased_after_evolution": true
}
```

触发条件：

```text
score_rate_after > score_rate_before + score_increase_threshold
```

当前默认阈值可继续沿用 `DEFAULT_SCORE_INCREASE_THRESHOLD = 0.05`。如需使用 0.02 或 0.03，应作为后续参数调整，不在第一批强改。

### 10.3 修正后续状态解释

当前 `update_sample_state.py` 中 `score_increased` 会返回：

```text
validated_high_score_sample
```

这与“得分升高是负收益”冲突。第一批应做最小修正：

```text
score_increased → continue_with_new_operator
```

并保持：

1. 写入 failure memory；
2. 不写 operator success memory；
3. 不进入 `stable_high_score_stop`；
4. 不进入 `validated_high_score_sample`；
5. 不立即强惩罚 operator，只建议切换 surface form 或 operator。

## 11. O1/O2/O4/O8 文本增量替换

第一批只修改以下现有字段：

```text
name
goal
required_question_shape
avoid
default_evaluation_focus
```

不得扩展 `OperatorPromptSpec`。

### 11.1 O1

当前风险：显式 A/B 缺口选择。

建议替换：

```text
name = 信息增量辨析

goal = 压测模型能否判断哪些题干事实真正改变结论强度，而不是把所有相关信息都等同看待。

required_question_shape = 围绕一个具体结论、处置边界或事实关系，让模型判断现有材料能支持到什么程度。不要显式要求在 A/B 缺口中选择，也不要直接点名“关键缺口”。

avoid = 不要把题面写成“哪个是最小关键事实/关键缺口”；不要把两个候选设计得一眼可分；不要让模型只需选择题面给出的明显答案。
```

### 11.2 O2

当前风险：显式层级定位。

建议替换：

```text
name = 事实—结论承接审查

goal = 压测模型能否判断题干事实与拟定结论之间是否能直接承接，而不是把初步线索直接上推为完整结论。

required_question_shape = 给出一个拟定结论、研判表述或处置判断，要求模型判断其是否能由题干事实支撑。不要显式要求指出“哪一层已成立、哪一层未成立”。

avoid = 不要出现明显层级提示；不要把题面写成教学式拆解；不要直接告诉模型结论应收窄到哪一步。
```

### 11.3 O4

当前风险：显式排序 / 分层。

建议替换：

```text
name = 结论依据取舍

goal = 压测模型能否区分哪些事实能直接支持结论，哪些只是背景、线索或弱相关信息。

required_question_shape = 围绕一个拟定结论或处置判断，让模型判断哪些题干事实可以作为直接依据，哪些只能作为背景或辅助线索。不要要求排序、分层或使用表格。

avoid = 不要显式要求按层级排序；不要把题面做成三类标签题；不要让格式成为主要难度。
```

### 11.4 O8

当前风险：显式双门槛。

建议替换：

```text
name = 结论强度边界

goal = 压测模型能否判断强结论是否被当前事实充分支撑，而不被显眼事实直接带到过强定性。

required_question_shape = 围绕一个可能过强的结论表述，让模型判断该表述是否能由题干事实支撑，或应如何限定。不要显式写出“双门槛、动作层、性质层”。

avoid = 不要点名双门槛结构；不要直接提示哪一部分已成立、哪一部分缺事实；不要把题面写成泛化的“是否稳妥”模板题。
```
