# 轻量回溯式多边界探索优化方案

## 1. 方案定位

本方案是在现有 Question_Evolution 链式机制基础上的轻量增强，不是重写当前项目，不是引入完整复杂树搜索，也不是把每个样本都展开成深层搜索树。

当前项目已经具备：

1. round0 稳定评分；
2. 样本画像与候选筛选；
3. operator 路由；
4. 多候选 question evolution；
5. 候选质量验证与 candidate selection；
6. 重新生成答案、rubric、评分；
7. 进化效果分析；
8. 状态更新和 memory 沉淀。

原始“树状搜索与回溯式多边界探索方案”的核心目标是正确的：一个样本命中一个边界后，不应马上整体停止，而应允许回到根节点或父节点，继续开新分支，最终让一个原始样本产出多条有效边界分支。fileciteturn41file0

但原方案涉及 `search_graph.jsonl`、`active_frontier.jsonl`、多节点状态、全脚本大范围改造、复杂停止条件和搜索预算。按照前面轻量化原则，当前不应一次性引入完整树搜索，否则项目会再次变复杂，甚至压过 question evolution 本身。

因此，本方案将原始方案压缩为：

```text
链式主流程 + 轻量回溯任务队列 + 多边界结果池
```

核心目标只有一个：

```text
命中一个有效边界后，不把样本整体停掉，而是基于根节点或父节点再开一条新分支，继续寻找其他边界。
```

---

## 2. 设计原则

### 2.1 不做完整树搜索

当前阶段不做：

1. 不构建完整通用搜索图；
2. 不要求每个节点都持久化完整树结构；
3. 不引入复杂图遍历；
4. 不让所有脚本都理解树节点；
5. 不按深树方式无限扩展；
6. 不改造现有 run_loop 的主体执行顺序。

### 2.2 只做轻量回溯

当前阶段只做：

1. 记录已命中的边界点；
2. 当某分支命中边界后，创建一个“回溯开新分支任务”；
3. 新分支可以从 root prompt 或 parent prompt 重新开始；
4. 新分支使用不同 boundary_axis 或不同 operator family；
5. 最终保留多个有效边界分支；
6. 搜索预算很小，避免成本失控。

### 2.3 保留现有主链机制

现有主链仍然保留：

```text
profile_samples.py
→ select_evolution_candidates.py
→ operator_router.py
→ question_evolution.py
→ validate_evolved_question.py
→ validate_difficulty_gain.py
→ candidate_selection.py
→ collect_answers.py
→ gen_rubric.py
→ scoring.py
→ analyze_evolution_effect.py
→ update_sample_state.py
```

新增机制只在 `update_sample_state.py` 之后产生下一轮的轻量 fork 任务，不要求重写所有阶段。

当前阶段的最终产物不是一棵复杂树，而是：

```text

```

即：

```json

  "root_prompt": "...",
  "discovered_boundaries": [
    {
      "boundary_id": "xxx_b1",
```

重点是“多个有效边界并存”，不是“所有中间节点都复杂建模”。

### 4.1 不新增完整 search_graph

原方案建议新增 `search_graph.jsonl`，字段较多。当前轻量版暂不新增完整搜索图。

第一版只新增两个轻量文件：

```text
boundary_bank.jsonl
fork_tasks.jsonl
```

### 4.2 boundary_bank.jsonl

用于记录已经发现的有效边界点。

每行记录一个 boundary：

```json
{

```

```json

    "root_id": "6406_root",
```

```json
    "is_backtrack_fork": true,
    "fork_reason": "previous_boundary_hit_open_new_axis"
  }
}
```

这比完整 `node_id / parent_node_id / active_frontier` 结构更轻量。

## 5. 回溯触发条件

### 5.1 主要触发：命中有效边界

当 `analyze_evolution_effect.py` 输出：

```text
effect_label = effective_boundary_probe
```

并且该边界不是重复边界时，触发回溯开新分支。

处理方式：

```text
记录 boundary_bank
→ 判断样本是否还有分支预算
→ 如果有，创建 fork_task
→ fork_task 从 root 或 parent 重新生成候选
```

### 5.2 可选触发：探索候选出现明显新失败模式

如果 exploration candidate 没有达到 effective boundary，但出现新的、可归因的目标 failure，也可以记录为弱边界候选。

第一版不强制触发 fork，只写入报告：

```text
probable_boundary_signal
```

### 5.3 不触发回溯的情况

以下情况不触发新分支：

1. `score_increased_after_evolution`
2. `no_clear_effect`
3. `pass_through`
4. `invalid_complexity`
5. `no_candidate_passed_quality_gate`
6. 与已有边界重复

这些只进入普通 memory，不创建 fork task。

## 6. 回溯到 root 还是 parent

### 6.1 默认回 root

第一版默认：

```text
命中边界后，优先从 root prompt 开新分支
```

原因：

1. root prompt 保留原始事实底座；
2. 不容易继承当前分支的题面偏差；
3. 更适合探索不同能力轴；
4. 实现简单。

### 6.2 何时回 parent

如果当前分支深度已经大于 1，且当前边界来自某个中间题的局部细化，可以从 parent 开兄弟分支。

第一版简化规则：

```text
如果 branch_metadata.source_node_type = parent
或 effect_analysis.recommended_next_action = fork_from_parent
则从 parent 开分支
否则从 root 开分支
```

但第一版不强制做复杂 parent 链追踪。若 parent prompt 不可恢复，就退回 root。

### 6.3 不做深层回溯

第一版不做：

1. 从任意祖先节点回溯；
2. 多层 parent stack；
3. 复杂 sibling expansion；
4. 路径级搜索。

只支持：

```text
root fork
parent fork
```

## 7. 分支预算设计

为了避免树搜索失控，第一版预算必须很小。

推荐配置：

```bash
ENABLE_BACKTRACK_FORK=true
MAX_BOUNDARIES_PER_SAMPLE=3
MAX_FORK_TASKS_PER_SAMPLE=3
MAX_FORK_TASKS_PER_ROUND=10
MAX_FORK_CANDIDATES_PER_TASK=1
MAX_FORK_DEPTH=1
```

### 7.1 含义

`MAX_BOUNDARIES_PER_SAMPLE`

一个原始样本最多保留多少个有效边界点。默认 3。

`MAX_FORK_TASKS_PER_SAMPLE`

一个样本最多开多少个回溯分支。默认 3。

`MAX_FORK_TASKS_PER_ROUND`

一轮最多执行多少个 fork task。默认 10。

`MAX_FORK_CANDIDATES_PER_TASK`

每个 fork task 生成几个候选。默认 1。

`MAX_FORK_DEPTH`

回溯分支第一版只扩展 1 层。默认 1。

### 7.2 为什么第一版只设 depth=1

当前目标不是深树搜索，而是：

```text
命中一个边界后，再从 root / parent 找另一个边界
```

因此每条回溯分支先只做一次进化即可。  
如果该分支也命中边界，再由下一轮继续开新分支。

## 8. 边界去重

### 8.1 为什么需要去重

回溯分支可能命中和已有分支相同的能力边界。此时不应作为新边界点记录。

### 8.2 轻量去重规则

第一版只用简单规则：

如果同时满足：

```text
same sample_id
and same boundary_axis
and same operator_used
and same surface_form_family
```

则视为可能重复。

如果再满足：

```text
prompt_new 与已有 boundary prompt 相似度较高
```

则标记为 duplicate。

### 8.3 不做复杂语义去重

第一版不做复杂 embedding 检索或 LLM dedup。  
可以用简单文本相似度，例如 token overlap / char ngram。

### 8.4 输出字段

```json
{
  "is_new_boundary_for_sample": true,
  "duplicate_of_boundary_id": null,
  "dedup_reason": ""
}
```

如果重复：

```json
{
  "is_new_boundary_for_sample": false,
  "duplicate_of_boundary_id": "6406_boundary_001",
  "dedup_reason": "same axis/operator/surface form and high prompt similarity"
}
```

## 9. fork task 如何进入现有流程

### 9.1 不重写 run_loop

第一版不重写主 run_loop。

只在每轮开始前增加一个步骤：

```text
merge fork_tasks into round input
```

也就是：

```text
上一轮 state_updated.jsonl
+
pending fork_tasks.jsonl 转成的样本 record
→ 本轮 input.jsonl
```

### 9.2 fork task 转成样本 record

fork task 转成普通样本 record 时，需要填充：

1. `sample_id`
2. `prompt`
3. `reference_answer`
4. `rubric`
5. `score_prompt`
6. `meta_info`
7. `branch_metadata`
8. `evolution_state`

其中 `prompt` 使用 `source_prompt`。

### 9.3 branch_metadata

```json
{
  "branch_metadata": {
    "is_backtrack_fork": true,
    "root_id": "6406_root",
    "branch_id": "6406_branch_002",
    "parent_boundary_id": "6406_boundary_001",
    "source_node_type": "root",
    "target_boundary_axis": "subclaim_localization",
    "fork_reason": "previous_boundary_hit_open_new_axis"
  }
}
```

### 9.4 operator_router 如何使用 fork task

`operator_router.py` 如果看到 `branch_metadata.target_boundary_axis`，应优先按该 axis 选择 operator。

如果当前已有 routing memory，也可以作为参考，但不要覆盖 fork task 的目标轴。

## 10. 如何选择新分支目标轴

### 10.1 不新增复杂 axis planner

第一版不新增独立 axis planner。

直接复用已有信息：

1. `sample_profile.core_capability`
2. `overscore_diagnosis.target_failure_mode`
3. `operator_route.backup_operators`
4. `difficulty_gain_validation.expected_failure_point`
5. 已发现 boundary 的 `boundary_axis`

### 10.2 目标轴选择原则

新分支应避开已命中的 boundary_axis，优先选择不同能力轴。

推荐顺序：

1. 如果已有 `operator_route.backup_operators`，选择与已命中 operator 不同的 backup。
2. 如果已有 `recommended_next_axes`，选择未命中过的 axis。
3. 如果没有，按默认 axis pool 选择。

### 10.3 默认 axis pool

```text
evidence_relation_discrimination
near_level_ranking
subclaim_localization
gap_choice
single_variable_counterfactual
fact_binding_constraint
```

避免优先使用：

```text
conclusion_strength_control_only
```

因为它容易退化成泛化审慎判断题。

## 11. 与三分流机制的关系

本方案默认沿用前面的轻量三分流机制：

```text
hard_reject
main_chain_candidate
exploration_candidate
```

### 11.1 main chain 命中边界

如果 main chain candidate 命中 `effective_boundary_probe`：

```text
记录 boundary
创建 fork task
```

### 11.2 exploration 命中边界

如果 exploration candidate 命中 `effective_boundary_probe`：

```text
也记录 boundary
也可以创建 fork task
```

这点很重要，因为 exploration 的目的就是让不确定候选通过真实评分证明价值。

### 11.3 hard reject 不参与 fork

hard reject 候选不创建 fork task，只写失败记录。

---

## 12. 与 score_increased 的关系

`score_increased_after_evolution` 不触发 fork。

处理：

1. 写入 failure memory；
2. 标记该 branch 失败；
3. 不创建新分支；
4. 但不停止整个 sample；
5. 如果样本还有预算，可以从 root 使用不同 axis 重新 fork。

也就是说：

```text
score increase = 当前分支失败
不是 sample 失败
```

---

## 13. 何时停止一个 sample

### 13.1 停止条件

一个 sample 整体停止，当满足任一条件：

1. 已发现边界数达到 `MAX_BOUNDARIES_PER_SAMPLE`；
2. fork task 数达到 `MAX_FORK_TASKS_PER_SAMPLE`；
3. 最近若干轮没有新增边界；
4. 所有可用 axis 都已尝试；
5. 样本预算耗尽；
6. 连续分支都出现 score increase 或 no gain。

### 13.2 不应停止的情况

以下情况不应停止整个 sample：

1. 某一条分支命中边界；
2. 某一条分支 score increase；
3. 某一条分支 no gain；
4. 某一轮没有候选通过；
5. 某个 operator 失败。

这些只影响当前分支或当前 operator + surface form。

## 14. 需要修改的脚本

### 14.1 update_sample_state.py

核心修改点：

1. 识别 `effective_boundary_probe`；
2. 写入 `boundary_bank.jsonl`；
3. 判断是否是新边界；
4. 如果还有预算，创建 `fork_tasks.jsonl`；
5. `score_increased_after_evolution` 只标记当前分支失败，不停止 sample；
6. 更新 sample 级预算字段。

### 14.2 operator_router.py

轻量修改：

1. 读取 `branch_metadata.target_boundary_axis`；
2. 优先按该 axis 选择 operator；
3. 避免重复已命中的 boundary_axis；
4. 不做复杂树搜索。

### 14.3 run_loop.sh

轻量修改：

每轮开始前增加：

```text
prepare_round_input_with_forks.py
```

作用：

```text
上一轮 state_updated.jsonl
+ pending fork_tasks.jsonl
→ 本轮 input.jsonl
```

### 14.4 新增 prepare_round_input_with_forks.py

职责：

1. 读取上一轮主链输出；
2. 读取 pending fork tasks；
3. 将 fork task 转成普通样本 record；
4. 合并去重；
5. 输出本轮 input.jsonl；
6. 标记 fork task 为 consumed 或 running。

### 14.5 analyze_evolution_effect.py

轻量修改：

1. 输出 `boundary_axis_detected`；
2. 输出 `is_new_boundary_for_sample` 可由 update_sample_state 二次确认；
3. 不要求复杂边界分类。

## 15. 不建议当前修改的内容

当前不建议做：

1. 不把所有脚本改成树节点处理器；
2. 不强制引入 `search_graph.jsonl`；
3. 不新增复杂 `active_frontier.jsonl`；
4. 不实现深度优先 / 广度优先 / UCB 等搜索算法；

7. 不引入多层 parent stack；
8. 不改写整个 run_loop。

这些可以作为后续高级版本，但当前目标是轻量回溯。

## 16. 测试设计

### 16.1 boundary hit 创建 fork

输入：

```text
effect_label = effective_boundary_probe
sample_budget_remaining > 0
已有 boundary_axis = evidence_relation_discrimination
```

期望：

1. 写入 boundary_bank；
2. 创建 fork_task；
3. fork_task 目标 axis 不等于已命中 axis。

### 16.2 boundary hit 不停止 sample

输入某分支命中边界。

期望：

```text
branch_status = boundary_hit
sample_status != stop_sample
```

除非预算已耗尽。

### 16.3 score increase 不创建 fork

输入：

```text
effect_label = score_increased_after_evolution
```

期望：

1. 写 failure memory；
2. 不创建 fork task；
3. 不停止 sample；
4. 当前 branch 标记 failed。

### 16.4 root fork task 转 record

输入 fork task。

期望输出 record：

1. prompt = source_prompt；
2. branch_metadata.is_backtrack_fork = true；
3. target_boundary_axis 存在；
4. 可以进入现有 pipeline。

### 16.5 每样本 fork 上限

当 sample 已达到 `MAX_FORK_TASKS_PER_SAMPLE`：

期望：

```text
不再创建新 fork task
```

### 16.6 每样本 boundary 上限

当 sample 已达到 `MAX_BOUNDARIES_PER_SAMPLE`：

期望：

```text
stop_sample
```

### 16.7 duplicate boundary

当新 boundary 与已有 boundary 同 axis、同 operator、同 surface form 且 prompt 相似：

期望：

```text
is_new_boundary_for_sample = false
不增加 discovered boundary count
```

## 18. 推荐实施顺序

### 第一步：boundary_bank

在 `update_sample_state.py` 中，当出现 `effective_boundary_probe` 时写入 `boundary_bank.jsonl`。

### 第二步：fork_tasks

命中边界后，如果样本还有预算，创建一个 fork task。

### 第三步：prepare_round_input_with_forks.py

将 pending fork tasks 转成普通 record，合并到下一轮 input。

### 第四步：operator_router 支持 target_boundary_axis

让 fork task 能按目标 axis 路由。

### 第五步：预算与停止条件

增加简单预算：

```text
MAX_BOUNDARIES_PER_SAMPLE
MAX_FORK_TASKS_PER_SAMPLE
MAX_FORK_TASKS_PER_ROUND
```

### 第六步：边界去重

先做轻量去重，不做复杂 embedding。

```text
请在当前 Question_Evolution 项目上实现轻量回溯式多边界探索机制。注意：不要重写现有主流程，不要引入完整复杂树搜索。目标是在现有链式 question evolution 基础上增加“命中一个边界后回到 root 或 parent 开新分支”的能力，使一个原始样本最终可以产出多个有效边界分支。

核心原则：
1. 不做完整 search_graph。
2. 不做复杂 active_frontier。
3. 不改写整个 run_loop。
4. 不把所有脚本改成树节点处理器。
5. 只新增轻量 boundary_bank.jsonl 和 fork_tasks.jsonl。
6. 第一版默认从 root 开新分支，parent fork 只做可选 fallback。
7. 每个 fork task 只生成少量候选，默认 1 个。
8. 命中 effective_boundary_probe 后记录 boundary，并在预算允许时创建 fork task。
9. score_increased_after_evolution 只表示当前分支失败，不创建 fork，不停止整个 sample。
10. sample 只有在边界数量、fork 数量或预算耗尽时才 stop。

需要实现：
1. update_sample_state.py：
   - 识别 effective_boundary_probe；
   - 写入 boundary_bank.jsonl；
   - 判断是否是新边界；
   - 若预算允许，创建 fork_tasks.jsonl；
   - score_increased_after_evolution 不得 stop sample；
   - 维护简单 sample 预算。

2. 新增 prepare_round_input_with_forks.py：
   - 读取上一轮 state_updated.jsonl；
   - 读取 pending fork_tasks.jsonl；
   - 将 fork task 转为普通样本 record；
   - 写入本轮 input.jsonl；
   - 标记 fork task consumed。

3. operator_router.py：
   - 如果 record 中有 branch_metadata.target_boundary_axis，优先按该 axis 路由；
   - 避免重复已命中过的 axis；
   - 不做复杂树搜索。

4. analyze_evolution_effect.py：
   - 尽量输出 boundary_axis_detected；
   - 保留 effect_label；
   - 不需要复杂分类器。

5. 配置项：
   - ENABLE_BACKTRACK_FORK=true
   - MAX_BOUNDARIES_PER_SAMPLE=3
   - MAX_FORK_TASKS_PER_SAMPLE=3
   - MAX_FORK_TASKS_PER_ROUND=10
   - MAX_FORK_CANDIDATES_PER_TASK=1
   - MAX_FORK_DEPTH=1

6. 测试：
   - effective_boundary_probe 会写 boundary_bank；
   - effective_boundary_probe 会创建 fork_task；
   - score_increased_after_evolution 不创建 fork，也不 stop sample；
   - fork_task 能转为普通 record；
   - 达到 boundary / fork 上限后不再创建新 fork；
   - duplicate boundary 不增加 boundary count。

验收标准：
1. 一个样本命中边界后不会立刻整体停止。
2. 系统可以从 root 或 parent 开新分支。
3. 一个样本可以记录多个 discovered boundaries。
4. 不引入完整复杂树搜索。
5. 主流程仍然基本保持原样。
6. 新机制通过小样本实验能看到 boundary_per_sample_avg 提升。
```

## 20. 最终原则

本方案不是为了把项目改成复杂搜索系统，而是为了在现有链式机制上补一个关键能力：

```text
命中一个边界后，回到 root / parent，换一个能力轴开新分支。
```

最终目标是：

```text
一个原始样本
→ 多条有效边界分支
→ 多个可复用能力边界题
```

而不是：

```text
一条链命中一个边界后整个样本停止。
```
