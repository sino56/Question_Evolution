# 方案二：轻量增强与记忆收敛实现方案

## 1. Context｜任务背景

### 1.1 项目阶段定位

`Question_Evolution` 项目的核心目标不是把题目机械写长，而是通过：

```text
候选探索 -> 真实评分 -> 后验反馈 -> 继续进化
```

把原始题目逐步改造成更难、更能暴露弱模型能力边界的题目。

本方案属于低风险核心闭环之后的轻量增强阶段。上一阶段已经解决的核心问题是：

```text
候选能否进入真实评分闭环
```

本方案继续解决的是：

```text
在不显著增加流程复杂度的前提下，如何让系统逐步利用失败经验收敛搜索空间
```

目标对象是当前 pipeline、后续开发者和实验分析流程。整理后的方案需要能直接作为增量开发任务说明、测试依据和验收清单使用。

本方案不是重写已有功能，也不是替换现有闭环。开发时必须先识别当前代码已经落地的能力，只补齐方案二尚未覆盖的轻量增强点。

### 1.2 当前主流程

当前推荐主流程仍保持：

```text
admitted_seed_samples.jsonl
-> round0_stability_probe.py
-> profile_samples.py
-> select_evolution_candidates.py
-> operator_router.py
-> question_evolution.py
-> validate_evolved_question.py
-> validate_difficulty_gain.py
-> candidate_selection.py
-> collect_answers.py
-> gen_rubric.py
-> scoring.py
-> analyze_evolution_effect.py
-> update_sample_state.py
```

本方案不重排主流程，只在现有阶段之间或现有阶段内部做增量增强。

### 1.3 当前代码审视结论

基于当前代码审视，已有能力和本方案差距如下：

| 模块 | 当前已有能力 | 与本方案的差距 |
|---|---|---|
| `candidate_selection.py` | 已有 `candidate_flow` 四分流：`hard_reject`、`main_chain_candidate`、`exploration_candidate`、`pass_through_candidate`；已有 exploration 每轮预算、`selected_for_exploration`、`score_cap`；已有量化模板简化 hard reject 条件 | 不应重做候选分流。只需要增量消费 `light_factual_check`、补充 `template_affordance_risk` soft penalty 和相关统计 |
| `analyze_evolution_effect.py` | 已输出 `score_increased_after_evolution`，并保留 `effect_label = score_increased` | 不应重做负收益语义。后续改动只需保证该语义不回退 |
| `update_sample_state.py` | `score_increased` 已转为 `continue_with_new_operator`，并进入 failure memory 语义 | 不应再把 score increase 改成成功终止态；只需保证新增 memory 逻辑与该语义兼容 |
| `operator_router.py` | 已读取 operator memory 和 failure memory，并可输出 `avoid_operators` | 现有逻辑更接近命中 failure memory 后规避 operator；尚未形成最近 N 轮窗口、`operator + surface_form_family` 粒度、`warn_only -> downrank -> avoid` 慢收敛 |
| `select_evolution_candidates.py` | `uncertain_low` 目前属于停止类状态 | `uncertain_low_probe` 尚未落地；应作为默认关闭的低预算探索开关增量加入 |
| `run_loop.sh` | 已按现有 12 步主流程串联 `operator_router.py`、`validate_evolved_question.py`、`validate_difficulty_gain.py`、`candidate_selection.py` 等阶段 | 不应重排主流程；若新增轻量检查，应尽量作为兼容字段或低侵入步骤接入 |
| 测试 | 已有 candidate flow、score increase 负收益、run loop 顺序等测试 | 新测试只覆盖新增行为，避免改写已有测试语义 |

因此，本方案的实现方式应是：

```text
保留已落地能力
-> 对缺口做小步增量补齐
-> 不重写主流程
-> 不改变已有字段语义
-> 用新增测试约束新增行为
```

### 1.4 增量开发原则

本方案继续坚持轻量与增量原则：

1. 已经落地的 `candidate_flow`、exploration 预算、`score_increased_after_evolution` 负收益语义不得重写；
2. 不把所有风险都提前变成 `hard_reject`；
3. 不默认引入复杂 LLM fact validator；
4. 不修改 `gen_rubric.py`、rubric prompt、score prompt 或 `scoring.py`；
5. failure memory 采用慢收敛，不因一次失败强惩罚；
6. template/rubric 风险主要用于排序、观察和报告；
7. 只有最近窗口内连续同类失败后，才影响 operator routing；
8. 不把项目做成 candidate rejection system，候选价值仍优先交给真实评分后验判断。

## 2. Request｜具体任务

请在现有代码基础上做“轻量增强与记忆收敛”的增量补齐。具体包括：

1. 先保留当前已经落地的四分流、exploration 预算、score increase 负收益语义和 run loop 主顺序；
2. 新增 `light_factual_check` 能力，逻辑位置在 `validate_evolved_question.py` 后、`validate_difficulty_gain.py` 前；
3. 让 `light_factual_check.fatal_errors` 非空时进入 `hard_reject`；
4. 让 `light_factual_check.warnings` 只影响排序分，不直接 hard reject；
5. 将 `template_affordance_risk` 作为 candidate selection 的 soft penalty，而不是替换现有 selection 公式；
6. 保留当前量化模板简化 hard reject 条件，仅在明确量化条件满足时触发；
7. 将 `rubric_shortcut_risk` 报告化，不作为门禁；
8. 为 failure memory 增加最近 N 轮窗口；
9. 在 `operator_router.py` 中按 `warn_only -> downrank -> avoid` 轻量消费 memory，但不粗暴替换现有 rule route；
10. 新增默认关闭的 `uncertain_low_probe` 开关；
11. 增加实验统计与小型 pytest 覆盖；
12. 保持历史 JSONL 字段兼容，不强制迁移旧数据；
13. 完成后运行可行验证，并输出自我审核报告。

## 3. Output Format｜交付结构

### 3.1 新增或消费字段

实现时建议保持字段可选、向后兼容。新增字段优先放入已有结构或稳定顶层字段中。

| 字段 | 建议位置 | 用途 | 兼容要求 |
|---|---|---|---|
| `light_factual_check` | 顶层字段，或紧跟 `validation_result` 的稳定字段 | 记录轻量事实检查结果 | 缺失时按通过处理，避免历史数据崩溃 |
| `template_affordance_risk` | `difficulty_gain_validation`、`validation_result` 或顶层兼容读取 | candidate selection soft penalty | 缺失时不扣分 |
| `surface_form_family` | `meta_info.question_evolution_metadata`、`candidate_generation` 或顶层兼容读取 | memory 同类失败匹配 | 缺失时用 `unknown`，不能因此 hard reject |
| `rubric_shortcut_risk` | 报告字段或分析统计字段 | 观察 rubric 套话风险 | 不参与 hard reject |
| `operator_route.memory_warnings` | `operator_route` | 1 次同类失败时提示 | 不改变主 operator |
| `operator_route.downrank_operator_surface_forms` | `operator_route` | 2 次同类失败时降权 | 只作用于 operator + surface form |
| `operator_route.avoid_operator_surface_forms` | `operator_route` | 3 次同类失败时规避 | 不粗暴封禁整个 operator |
| `effect_analysis.score_increased_after_evolution` | `effect_analysis` | 标记得分升高的负收益 | 已有语义不得回退 |

`light_factual_check` 推荐结构：

```json
{
  "light_factual_check": {
    "passed": true,
    "fatal_errors": [],
    "warnings": [],
    "risk_tags": []
  }
}
```

`template_affordance_risk` 推荐结构：

```json
{
  "template_affordance_risk": "low / medium / high",
  "surface_form_family": "generic_cautious_judgment / evidence_relation_comparison / counterfactual_boundary / fact_binding_review / unknown",
  "template_risk_reason": "简要说明风险"
}
```

### 3.2 流程插入点

逻辑位置应为：

```text
validate_evolved_question.py
-> light_factual_check
-> validate_difficulty_gain.py
-> candidate_selection.py
```

实现形态优先选择低侵入方式：可以是在现有验证阶段后补充字段，也可以新增独立轻量脚本，但不要为了该检查重写 `validate_evolved_question.py` 或 `validate_difficulty_gain.py` 的主逻辑。无论采用哪种形态，都必须保证：

1. 高置信 fatal error 在 selection 层可以被 hard reject；
2. warning 只能扣分或进入报告；
3. `run_loop.sh` 的整体顺序不被重排；
4. 历史缺字段数据仍可走 fallback；
5. 已有 `candidate_flow` 和 `selection_status` 语义不被替换。

## 5. 具体设计

### 5.1 light_factual_check

`light_factual_check` 是低成本规则检查，不是完整事实核验系统。第一版只拦截高置信事实错误。

hard reject 只包括：

1. 数字、时间、金额、距离、比例新增或冲突；
2. 原题明确实体名被替换；
3. 候选题出现原题没有的确定性证据结果；
4. 候选题明确修改原题核心数值；
5. 候选题明确修改原题核心实体；
6. 明确文本级反转，例如原题要求“不得补充新材料”，候选题改成“可以补充新材料”。

只 warning 的情况：

1. 一般场景收窄；
2. 处置边界限制；
3. 可用材料限制；
4. 结论措辞变化；
5. 背景信息轻微改写；
6. 删除关键限制但未出现明确文本级反转。

candidate selection 使用方式：

```text
fatal_errors 非空 -> hard_reject
warnings 非空 -> selection_score - 0.05
```

为什么这样做：事实错误会污染后续 difficulty gain 判断，因此高置信 fatal error 可以前置拦截；但轻量规则无法可靠判断复杂语义变化，所以 warning 不能直接压死候选。

### 5.2 template_affordance_risk soft penalty

`template_affordance_risk` 只作为排序信号。

默认扣分：

```text
template_affordance_risk = medium -> selection_score - 0.05
template_affordance_risk = high -> selection_score - 0.10
```

hard reject 例外：

```text
generic_answer_success_likelihood >= 0.85
and specific_fact_dependency_score <= 0.35
```

只有同时满足以上两个量化条件时，模板风险才可触发 hard reject。如果没有这两个字段，不得仅凭主观模板判断 hard reject。

为什么这样做：模板化题面确实可能让弱模型靠套话得分，但这类风险不一定等价于坏题。先作为排序信号，能避免系统重新退化成前置拒绝器。

### 5.3 rubric_shortcut_risk 报告化

`rubric_shortcut_risk` 当前阶段只进入报告和统计，不参与 hard reject。

不修改以下模块：

1. `gen_rubric.py`
2. rubric prompt
3. score prompt
4. `scoring.py`

观察目标：

```text
候选题是否容易诱导后续 rubric 奖励套话
```

如果该风险在实验中频繁出现，后续可单独进入 rubric 质量阶段处理，但不在本方案解决。

### 5.4 failure memory 最近 N 轮窗口

如果 memory 永久生效，早期失败会长期污染路由，导致搜索空间过早收缩。因此需要最近 N 轮窗口。

默认配置：

```text
FAILURE_MEMORY_WINDOW_ROUNDS = 3
```

可配置为：

```text
3 或 5
```

同类失败必须同时匹配：

```text
sample_signature
+ operator_used
+ surface_form_family
+ failure_type
```

字段兼容关系：

| 语义 | 主字段 | 兼容别名 |
|---|---|---|
| operator | `operator_used` | `operator_id`, `candidate_operator` |
| sample signature | `sample_signature` | `sample_profile_signature` |
| score before | `score_rate_before` | `score_before` |
| score after | `score_rate_after` | `score_after` |
| score delta | `delta_score_rate` | `score_delta` |
| surface form | `surface_form_family` | `question_surface_form` |
| failure type | `failure_type` | `effect_label` |

慢收敛规则：

```text
最近 N 轮内同类失败 1 次 -> warn_only
最近 N 轮内同类失败 2 次 -> downrank operator + surface form
最近 N 轮内同类失败 3 次 -> avoid operator + surface form 或切 operator family
```

惩罚对象是：

```text
operator_used + surface_form_family
```

不是整个 operator。

### 5.5 operator_router 轻量消费 memory

1 次同类失败时，只输出警告：

```json
{
  "operator_route": {
    "memory_warnings": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "failure_count": 1,
        "action": "warn_only"
      }
    ]
  }
}
```

2 次同类失败时，降权该 operator 与 surface form 组合：

```json
{
  "operator_route": {
    "downrank_operator_surface_forms": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "reason": "repeated_negative_gain"
      }
    ]
  }
}
```

3 次同类失败时，规避该 operator 与 surface form 组合：

```json
{
  "operator_route": {
    "avoid_operator_surface_forms": [
      {
        "operator_used": "O1_gap_choice",
        "surface_form_family": "generic_cautious_judgment",
        "reason": "repeated_negative_gain"
      }
    ]
  }
}
```

为什么这样做：路由应根据具体失败形态慢慢收敛，而不是因为某个 operator 的一次失败就封禁整个能力轴。

### 5.6 uncertain_low_probe

`uncertain_low_probe` 是探索开关，不是默认主流程。

默认关闭：

```text
ENABLE_UNCERTAIN_LOW_PROBE = false
```

开启条件：

```text
admission_status = uncertain_low
and overscore_diagnosis.is_worth_evolving = true
and stable_score >= UNCERTAIN_LOW_PROBE_MIN_SCORE
```

默认阈值：

```text
UNCERTAIN_LOW_PROBE_MIN_SCORE = 0.55
```

预算：

```text
recommended_evolution_budget = 1
```

为什么这样做：低预算探索可以扩大样本覆盖面，避免系统长期只进化少数 stable_high 或 borderline 样本；默认关闭可以控制实验风险。

### 5.7 实验指标增强

方案实施后，建议每轮新增统计：

1. `light_factual_fatal_count`
2. `light_factual_warning_count`
3. `template_affordance_risk_distribution`
4. `rubric_shortcut_risk_distribution`
5. `failure_memory_write_count`
6. `failure_memory_warn_only_count`
7. `failure_memory_downrank_count`
8. `failure_memory_avoid_count`
9. `operator_surface_form_failure_distribution`
10. `uncertain_low_probe_count`
11. `main_chain_selection_rate`
12. `exploration_selection_rate`
13. `any_scoring_entry_rate`
14. `effective_boundary_probe_count`
15. `score_increased_after_evolution_count`

核心观察：

1. `light_factual_check` 是否误杀候选；
2. template 风险是否只是影响排序，而不是压死通过率；
3. memory 是否开始有效收敛；
4. router 是否过早缩小搜索空间；
5. `uncertain_low_probe` 是否扩大探索面。

## 6. 实施顺序

### 第 0 步：前置确认

实施前先确认低风险核心闭环仍满足，并把已满足项视为“保留对象”而不是“重做对象”：

1. `candidate_selection.py` 已支持 `hard_reject`、`main_chain_candidate`、`exploration_candidate`、`pass_through_candidate`；
2. `candidate_selection.py` 已支持 `MAX_EXPLORATION_CANDIDATES_PER_ROUND` 和 `--max-exploration-candidates-per-round`；
3. `exploration_candidate` 已通过 `selected_for_exploration = true` 进入后续真实评分链路；
4. `analyze_evolution_effect.py` 已输出 `score_increased_after_evolution`；
5. `update_sample_state.py` 已避免把 `score_increased` 转成 terminal success；
6. `run_loop.sh` 已串联现有 12 步主流程；
7. O1/O2/O4/O8 显式脚手架减少相关测试应继续保留。

如果前置能力回退，应先修复回退问题，再实施本方案。不要把已落地功能作为方案二重新实现一遍。

### 第 1 步：实现 light_factual_check

先只做高置信规则，不做复杂语义判断。完成后对照检查：

1. fatal error 是否只覆盖明确事实错误；
2. warning 是否没有触发 hard reject；
3. 输出字段是否兼容历史数据；
4. 是否没有改变 `validate_difficulty_gain.py` 的既有 label schema。

### 第 2 步：接入 candidate_selection soft penalty

在现有 `score_candidate()` 和 `candidate_flow_info()` 基础上增量接入 factual warning 和 template risk，不替换已有 selection 公式。完成后对照检查：

1. `warnings` 只扣 0.05；
2. template medium 扣 0.05；
3. template high 扣 0.10；
4. 只有量化模板简化条件触发 hard reject；
5. rubric risk 不参与 hard reject；
6. `weak_gain`、`needs_manual_review`、`no_gain` 既有 exploration 规则不回退。

### 第 3 步：实现 failure memory 最近 N 轮窗口

先统计同类失败，不急于强改 router。完成后对照检查：

1. 窗口默认 N=3；
2. 只统计最近 N 轮；
3. 同类匹配使用 `sample_signature + operator_used + surface_form_family + failure_type`；
4. 缺字段不导致 hard reject；
5. 旧 memory 缺少 `surface_form_family` 时仍能以兼容 fallback 运行，但不得因此扩大惩罚范围。

### 第 4 步：operator_router 消费 memory

在现有 `_base_rule_route()`、operator memory 和 recommended next methods 逻辑之上分层接入 `warn_only -> downrank -> avoid`。完成后对照检查：

1. 1 次同类失败只 warning；
2. 2 次才 downrank；
3. 3 次才 avoid；
4. 不因一个 surface form 失败禁用整个 operator；
5. 已有 `avoid_operators` 输出兼容保留，不被新字段替代。

### 第 5 步：实现 uncertain_low_probe 开关

默认关闭，只在实验时打开。完成后对照检查：

1. 默认 `false` 时 `uncertain_low` 仍 stop；
2. 开启且满足条件时 `recommended_evolution_budget = 1`；
3. 不影响已准入样本的主流程；
4. 不改变 `select_evolution_candidates.py` 对其他 stop status 的既有解释。

### 第 6 步：增加统计与测试

补充每轮统计和 pytest 覆盖。完成后运行必要验证命令。

### 第 7 步：小样本实验

观察通过率、score drop、score increase、memory 使用情况，再决定是否进一步增强。

## 7. 测试要求

至少新增或更新测试覆盖。测试原则是：新增行为新增测试，已有行为只做防回退断言，不为了方案二重写旧测试。

1. `light_factual_check.fatal_errors` 非空时 hard reject；
2. `light_factual_check.warnings` 非空时只扣分，不 hard reject；
3. template medium 扣 0.05；
4. template high 扣 0.10；
5. template high 但没有量化字段时不 hard reject；
6. `generic_answer_success_likelihood >= 0.85` 且 `specific_fact_dependency_score <= 0.35` 时 hard reject；
7. `rubric_shortcut_risk` 只写报告，不参与 hard reject；
8. 最近 N 轮内同类失败 1 次 -> `warn_only`；
9. 最近 N 轮内同类失败 2 次 -> `downrank`；
10. 最近 N 轮内同类失败 3 次 -> `avoid`；
11. `operator_router` 不因单个 surface form 失败而禁用整个 operator；
12. `ENABLE_UNCERTAIN_LOW_PROBE=false` 时 `uncertain_low` 仍 stop；
13. `ENABLE_UNCERTAIN_LOW_PROBE=true` 且满足条件时，budget=1；
14. `score_increased_after_evolution` 语义不回退；
15. 已有 candidate flow、score increase、operator anti-scaffold 测试继续通过；
16. 已有 `run_loop.sh` 阶段顺序测试继续通过；
17. 历史缺字段样本不会因为缺少方案二新增字段而失败。

推荐验证命令：

```bash
pytest -q
```

如果只整理文档、没有修改代码，可至少运行：

```bash
git diff --check
```

## 8. Checkpoint｜暂停条件

除非出现以下情况，否则不要中途停止实施：

1. 需要删除文件、重置分支、提交代码、部署或覆盖大型实验结果；
2. 发现当前代码结构与方案关键前提严重冲突，继续实现会破坏主流程；
3. 缺少必须由用户提供的文件、数据、账号或实验目标；
4. 新增字段会与现有稳定字段发生不可兼容冲突；
5. 测试显示 candidate selection 通过率明显崩塌，且无法用轻量修正恢复；
6. 需要修改 `gen_rubric.py`、rubric prompt、score prompt 或 `scoring.py` 才能继续；
7. 需要重写已经落地的 candidate flow 或 score increase 负收益逻辑才能继续。

其他情况下，应基于现有信息做合理 fallback，并在最终报告中说明处理依据和不确定性。

## 9. Acceptance Criteria｜验收标准

完成后应满足：

1. `light_factual_check` 只拦高置信 fatal error；
2. factual warning 不会 hard reject；
3. template risk 和 rubric risk 不会单独压死候选；
4. 只有明确量化模板简化条件才触发 template hard reject；
5. failure memory 有最近 N 轮窗口；
6. 同类失败 1 次只记录或 warning；
7. 同类失败 2 次才 downrank；
8. 同类失败 3 次才 avoid；
9. router 不会过度惩罚整个 operator；
10. `uncertain_low_probe` 可配置开启，默认关闭；
11. 整体 candidate selection 通过率没有明显崩塌；
12. `score_increased_after_evolution` 仍被视为负收益；
13. 不修改 rubric 和 scoring；
14. 不强制迁移历史 JSONL；
15. 已有 candidate flow、score increase、run loop 顺序等行为不回退；
16. 新增或相关 pytest 通过；
17. 最终报告说明修改内容、对照方案检查结果、验证命令及剩余风险。
