# 第二阶段：候选题难度收益验证改造方案

## 1. 背景与问题

第一阶段已经完成 round0 稳定评分优化：系统不再依赖单次 Qwen 作答和单次 judge 评分，而是通过多 trial 形成 `round0_score_summary`，并将顶层 `score_rate` 重定义为 `stable_score`。这解决了“进化准入基线不稳定”的问题。

但 round0 稳定后，系统仍然存在第二个核心风险：候选题被生成出来以后，并没有强制证明它真的比原题更有压测价值。

当前候选题质量不稳定主要表现为：

1. 有时 GPT 能真正压边界，生成能暴露 Qwen 关键判断短板的新题。
2. 有时 GPT 会把原本应该由模型推理出来的答案线索写进题面，使题目更清楚、更容易。
3. 有时题目看起来更复杂，但难度主要来自格式要求、任务变多、题干变长，而不是来自真实专业判断。
4. 有时候选题仍然考原来的开放表达能力，没有制造新的竞争判断。
5. 有时候选题通过复杂度/可回答性校验，但 Qwen 仍然满分或得分升高。
6. 候选题被采用前，没有独立判断“是否减少线索泄漏、是否比原题更难、是否能稳定区分强弱模型”。

因此，第二阶段的目标是：在候选题正式进入后续参考答案生成、rubric 生成和重新评分之前，增加一个“难度收益验证门禁”。

---

## 2. 第二阶段目标

本阶段目标不是继续优化 `question_evolution.py` 的生成 prompt，而是在候选题采纳前增加独立验证。

具体目标：

1. 判断候选题是否保持原题核心主轴。
2. 判断候选题是否没有把关键答案线索写进题面。
3. 判断候选题是否没有因为“更清楚”而变得更容易。
4. 判断候选题是否制造了真实竞争判断。
5. 判断候选题是否没有把难度建立在格式复杂度、篇幅长度或多任务堆叠上。
6. 判断候选题是否仍然可由题干和参考材料回答。
7. 判断候选题是否预计能让 Qwen 在目标 failure mode 上失分。
8. 在条件允许时，对候选题执行弱模型 probe，估计是否真的带来 score drop。
9. 将验证结果写入候选题字段，供 `candidate_selection.py` 使用。
10. 不通过难度收益验证的候选题不得被正式采用。

---

## 3. 非目标

本阶段不解决以下问题：

1. 不重新设计 round0 稳定评分。
2. 不做完整失败回退树搜索。
3. 不改变 Qwen 正式评分逻辑。
4. 不重写所有算子 prompt。
5. 不强制要求每个候选题都重新生成最终 rubric。
6. 不替代后续 `analyze_evolution_effect.py` 的真实效果分析。

本阶段只做“候选题采纳前的独立门禁”。

当前大致流程是：

```text

  ↓

  ↓

  ↓
question_evolution.py
  ↓

  ↓

  ↓
collect_answers.py
  ↓
gen_rubric.py
  ↓
scoring.py
  ↓

```

第二阶段建议插入：

```text
question_evolution.py
  ↓
validate_evolved_question.py
  ↓
validate_difficulty_gain.py
  ↓
candidate_selection.py
```

也就是说，`validate_evolved_question.py` 负责判断“候选题是否合法、可回答、复杂度不过载”；新增的 `validate_difficulty_gain.py` 负责判断“候选题是否比原题更难、更少泄漏、更能制造边界判断”。

二者职责不同，不能合并：

| 模块 | 核心问题 |
|---|---|
| `validate_evolved_question.py` | 题目是否可回答、单主轴、复杂度不过载 |
| `validate_difficulty_gain.py` | 题目是否比原题更难、更少泄漏、更能制造边界判断 |
| `candidate_selection.py` | 在通过验证的候选中选择最佳候选 |

## 5. 核心原则

第二阶段必须遵守以下原则：

1. 候选题不能因为“把问题说得更清楚”而让弱模型更容易。
2. 候选题不能直接点名原题中的关键缺口、关键前提或正确答案方向。
3. 候选题不能把推理路径拆成明显步骤，导致 Qwen 只需顺着题面填空。
4. 候选题必须保留原题核心领域和能力主轴。
5. 候选题应制造一个明确的竞争判断，而不是泛泛要求“分析更全面”。
6. 候选题的难度应来自判断边界，而不是格式、篇幅、候选项数量或复杂输出结构。
7. 候选题必须仍然可回答，不能引入题外法律知识、侦查经验或未提供事实。
8. 通过验证只表示候选题有采纳资格，不表示最终一定有效；最终效果仍由后续评分闭环确认。

## 7. 难度收益验证维度

建议从 6 个维度评分，每个维度 0–1 分。

### 7.1 axis_consistency_score

判断候选题是否仍然考原题核心能力。

高分条件：

1. 保留原题领域和核心事实。
2. 没有转移到新任务。
3. 没有把犯罪分析问题改成泛泛法律常识题。
4. 没有引入题外材料。
5. 仍然对应 `sample_profile.core_capability` 或 `operator_route` 指向的能力轴。

低分风险：

1. 题目换了主轴。
2. 从证据判断变成写作组织。
3. 从案件事实推理变成法律条款背诵。
4. 从边界判断变成开放建议。

### 7.2 no_leakage_score

判断候选题是否没有泄漏答案线索。

高分条件：

1. 没有直接点名正确缺口。
2. 没有直接提示正确判断层级。
3. 没有把强模型答案中的关键结论写进题面。
4. 没有让错误选项明显弱于正确选项。
5. 没有把推理路径脚手架化。

低分风险：

1. 题面问“还缺哪条关键证据”，但关键证据已经写出来。
2. 二选一候选项中一个明显是废话，一个明显是正确答案。
3. 题面直接告诉模型“不能直接定性，只能停留在线索层面”。
4. 题面把“闭环未形成”“排他性不足”等答案方向提前暴露。

### 7.3 competitive_judgment_score

判断候选题是否制造了真实竞争判断。

高分条件：

1. 候选题要求模型在两个相近但不等价的判断之间选择。
2. 候选题要求区分线索、嫌疑、结论、排他性成立等层级。
3. 候选题要求判断哪个补强事实更关键，而不是简单罗列所有事实。
4. 候选题要求处理单变量反事实、伪闭环、近似项分层等。
5. 主要失分点集中在一个判断上。

低分风险：

1. 只是要求“更全面分析”。
2. 只是要求“说明理由”。
3. 只是给题目加了更多背景。
4. 只是把开放题改成边界题，但没有产生竞争选项。
5. 同时考多个能力点，导致失分不可归因。

### 7.4 anti_clarity_trap_score

判断候选题是否避免“题变清楚但更容易”。

高分条件：

1. 候选题没有把原本需要模型自行发现的判断点说破。
2. 候选题没有显式提示“本题重点是区分 A 与 B”。
3. 候选题没有将原本开放的隐藏难点拆成明显步骤。
4. 候选题没有变成教学式问题。
5. 弱模型不能仅靠顺着题面关键词就答对。

低分风险：

1. 原题难点是“发现还缺什么”，候选题直接问“还缺什么”。
2. 原题难点是“不能从异常直接上升到结论”，候选题直接要求“说明为什么不能直接定性”。
3. 原题难点是“识别哪个补强事实关键”，候选题直接标明“关键补强事实”。
4. 题面把答案结构提前列好。

### 7.5 answerability_score

判断候选题是否仍然可回答。

高分条件：

1. 题干事实足够支撑答案。
2. 不依赖题外法律条文。
3. 不依赖未提供的侦查结果。
4. 不要求模型进行现实世界事实查询。
5. 不引入参考答案无法覆盖的新知识。

低分风险：

1. 加入新事实后需要外部法律评价。
2. 题干不足以判断唯一答案。
3. 需要知道具体司法解释、证据规则或程序规范。
4. 候选项都可能成立，无法稳定评分。

### 7.6 format_complexity_score

判断难度是否没有来自格式复杂度。

高分条件：

1. 输出任务最多 1–2 个。
2. 不要求复杂表格。
3. 不要求多层标签体系。
4. 不靠字数、编号、格式限制制造难度。
5. 题干长度没有明显膨胀。

低分风险：

1. 题干过长。
2. 子问题过多。
3. 要求多维表格。
4. 要求固定句式、固定编号、复杂格式。
5. 答题困难主要来自遵循格式，而不是专业判断。

## 8. 线索泄漏风险标签

建议 `validate_difficulty_gain.py` 输出以下 `risk_tags`。

### 8.1 missing_premise_named

题面直接点名关键缺口或关键前提。

例如：

```text
请说明为什么“缺少排他性证据”导致无法定性。
```

如果原本难点就是让模型发现“缺少排他性证据”，那么这属于泄漏。

### 8.2 conclusion_hint_revealed

题面直接暗示正确结论方向。

例如：

```text
在不能直接认定嫌疑人作案的情况下……
```

如果原题本来需要判断是否能认定，那么这句话已经提示答案。

### 8.3 option_gap_too_obvious

候选项差距过大，正确项明显更专业或更贴近答案。

例如：

```text
A. 继续调查关键证据
B. 多写一些分析过程
```

这类二选一没有真实竞争判断。

### 8.4 layer_boundary_exposed

题面直接暴露应该区分的层级。

例如：

```text
请区分“异常线索”和“定案结论”。
```

如果原本难点就是发现不能从线索上升到结论，那么这会降低难度。

### 8.5 answer_path_scaffolded

题面给出完整推理路径，只让模型填充。

例如：

```text
先判断证据是否闭环，再判断是否排除其他可能，最后说明结论是否成立。
```

如果这些步骤本来应由模型自行组织，就属于脚手架化。

### 8.6 overclarified_prompt

候选题比原题更清楚，弱模型更容易顺着答。

这是第二阶段最重要的拦截对象之一。

### 8.7 format_difficulty_only

候选题难度主要来自格式或输出约束。

### 8.8 external_fact_dependency

候选题引入题外知识依赖。

### 8.9 multi_axis_overload

候选题混入多个主轴，后续失分不可归因。

### 8.10 no_real_gain

候选题没有明显难度收益，只是改写或扩写。

## 9. difficulty_gain_label

建议输出以下标签：

| 标签 | 含义 |
|---|---|
| `clear_gain` | 明确比原题更有压测价值 |
| `probable_gain` | 大概率有收益，但存在轻微风险 |
| `weak_gain` | 收益较弱，不建议优先采用 |
| `no_gain` | 没有明显收益 |
| `leakage_or_simplification` | 泄漏线索或题变简单 |
| `format_complexity_only` | 难度主要来自格式复杂度 |
| `axis_shift` | 偏离原题主轴 |
| `unanswerable_or_external` | 不可回答或依赖外部知识 |
| `needs_manual_review` | 自动判断不稳，需要人工复核 |

推荐通过条件：

```text
clear_gain 或 probable_gain 可以通过
weak_gain 默认不通过，除非没有其他候选
no_gain / leakage_or_simplification / format_complexity_only / axis_shift / unanswerable_or_external 一律不通过
needs_manual_review 默认不通过，可进入人工复核池
```

## 10. difficulty_gain_score 计算

建议由 LLM validator 输出 6 个维度分数，再用规则聚合。

```text
difficulty_gain_score =
  0.20 * axis_consistency_score
+ 0.25 * no_leakage_score
+ 0.20 * competitive_judgment_score
+ 0.15 * anti_clarity_trap_score
+ 0.10 * answerability_score
+ 0.10 * format_complexity_score
```

硬性拒绝条件优先于总分：

```text
no_leakage_score < 0.60
anti_clarity_trap_score < 0.60
answerability_score < 0.70
axis_consistency_score < 0.70
format_complexity_score < 0.50
risk_tags contains external_fact_dependency
risk_tags contains answer_path_scaffolded
risk_tags contains conclusion_hint_revealed
risk_tags contains missing_premise_named
```

推荐通过阈值：

```text
difficulty_gain_score >= 0.75
and no hard reject
```

边界通过：

```text
0.65 <= difficulty_gain_score < 0.75
and risk_tags 不包含高危标签
and candidate_generation 中没有更优候选
```

边界通过时应标记：

```json
"recommended_action": "admit_if_no_better_candidate"
```

## 11. LLM 验证 prompt 设计

新增 `prompts/difficulty_gain_validation.py` 或在脚本内定义模板。

### 11.1 输入内容

validator 应读取：

1. 原题 `original_prompt`
2. 候选题 `evolved_prompt`
3. 参考答案 `reference_answer`
4. round0 代表性 Qwen 答案 `representative_round0_answer`
5. 原 rubric
6. sample_profile
7. overscore_diagnosis
8. operator_route
9. candidate_operator
10. question_evolution_metadata
11. validate_evolved_question 的结果

### 11.2 输出 JSON

必须输出合法 JSON：

```json
{
  "passed": true,
  "difficulty_gain_label": "clear_gain",
  "dimension_scores": {
    "axis_consistency_score": 0.9,
    "no_leakage_score": 0.85,
    "competitive_judgment_score": 0.8,
    "anti_clarity_trap_score": 0.75,
    "answerability_score": 0.9,
    "format_complexity_score": 0.95
  },
  "risk_tags": [],
  "leakage_analysis": "说明是否泄漏答案线索。",
  "clarity_trap_analysis": "说明是否把题变清楚但更容易。",
  "competitive_judgment_analysis": "说明新题制造了什么真实竞争判断。",
  "expected_qwen_failure_match": true,
  "expected_failure_point": "Qwen 可能仍会把异常线索上推成结论成立。",
  "reject_reason": "",
  "recommended_action": "admit_candidate"
}
```

### 11.3 prompt 核心约束

validator 不能只看候选题是否“更详细”，必须比较原题与候选题：

1. 候选题是否泄漏了原题中的隐含难点。
2. 候选题是否把答案路径显式写出。
3. 候选题是否真正制造竞争判断。
4. 候选题是否仍然考同一个能力轴。
5. 候选题是否能让原 Qwen 答案的短板继续暴露。
6. 候选题是否只是更清楚、更格式化、更长。

## 12. 弱模型 probe 设计

仅靠 LLM validator 判断题面仍然不够。建议增加可选弱模型 probe。

### 12.1 probe 目标

在候选题被正式采用前，让 Qwen 对候选题快速作答，并用临时评分方式判断候选题是否可能带来难度收益。

### 12.2 probe 执行模式

#### light 模式

只让 Qwen 作答一次，不生成正式新 rubric。使用 LLM validator 对 Qwen probe answer 进行目标 failure 判断。

输出：

```json
"weak_probe": {
  "enabled": true,
  "mode": "light",
  "probe_answer": "...",
  "probe_failure_detected": true,
  "target_failure_match": true,
  "probe_judgment": "candidate likely exposes target failure"
}
```

#### score 模式

为候选题生成临时 rubric 或使用参考答案映射 rubric，对 Qwen probe answer 进行快速评分。

输出：

```json
"weak_probe": {
  "enabled": true,
  "mode": "score",
  "probe_score_rate": 0.68,
  "baseline_stable_score": 0.82,
  "estimated_score_drop": 0.14,
  "target_failure_match": true
}
```

### 12.3 第一版建议使用 light 模式

第一版可以只做：

```text
LLM validator 判断题面收益
+ 可选 Qwen light probe 判断是否命中目标 failure
```

第二版再加入 score mode。

## 13. weak_probe 通过规则

如果启用 weak probe，建议通过条件：

```text
difficulty_gain_validation.passed == true
and (
  weak_probe.disabled
  or weak_probe.target_failure_match == true
  or weak_probe.probe_judgment in {"candidate likely exposes target failure", "candidate is harder than original"}
)
```

如果 probe 显示：

```text
Qwen 直接答中关键点
或 Qwen 得分预计不下降
或 Qwen 没有暴露目标 failure
```

则候选题不一定直接拒绝，但应降低选择优先级：

```json
"recommended_action": "admit_low_priority"
```

如果候选题同时存在轻微泄漏风险和 weak probe 未命中，则直接拒绝。

## 14. candidate_selection.py 改造

`candidate_selection.py` 必须消费 `difficulty_gain_validation`。

### 14.1 候选过滤

候选题进入选择前必须满足：

```python
validation_result.passed is True
difficulty_gain_validation.passed is True
```

如果没有 `difficulty_gain_validation`，默认不通过，除非使用 legacy 参数：

```bash
--allow-missing-difficulty-gain
```

该参数仅用于兼容旧实验，不建议默认开启。

### 14.2 候选排序

候选得分建议改为：

```text
candidate_selection_score =
  0.45 * difficulty_gain_score
+ 0.20 * no_leakage_score
+ 0.15 * competitive_judgment_score
+ 0.10 * weak_probe_score
+ 0.10 * validation_quality_score
```

其中：

```text
weak_probe_score = 1.0 if target_failure_match else 0.5 if disabled else 0.0
```

### 14.3 高危标签一票否决

如果 `risk_tags` 包含：

```text
missing_premise_named
conclusion_hint_revealed
answer_path_scaffolded
external_fact_dependency
format_difficulty_only
axis_shift
```

候选题直接淘汰。

### 14.4 没有候选通过时的处理

如果一个 candidate group 中没有任何候选通过难度收益验证：

1. 不应随便选择最高分候选。
2. 应输出 pass-through 或失败标记。
3. 写入：

```json
"candidate_selection": {
  "selected": false,
  "selection_status": "no_candidate_passed_difficulty_gain",
  "recommended_next_action": "retry_with_backup_operator"
}
```

后续第三阶段失败回退机制可以读取该状态。

## 22. 验收标准

第二阶段完成后，应满足：

1. 新增 `validate_difficulty_gain.py`。
2. 每个候选题都有 `difficulty_gain_validation` 字段。

4. 高危风险标签一票否决。
5. 候选题如果把答案线索写进题面，会被拒绝。
6. 候选题如果只是格式变复杂，会被拒绝。
7. 候选题如果偏离原题主轴，会被拒绝。
8. 候选题如果不可回答或依赖题外知识，会被拒绝。
9. 候选题如果没有真实竞争判断，会被降权或拒绝。
10. 每轮输出 difficulty gain 统计报告。
11. report 中能统计每个 operator 的通过率和主要失败原因。
12. `run_loop.sh` 中 `candidate_selection.py` 之前必须插入 `validate_difficulty_gain.py`。
13. 如果一个 group 没有候选通过，不能随便选一个候选继续后续流程。
14. 测试覆盖泄漏、变简单、格式复杂、主轴偏移、不可回答、无候选通过等情况。
15. 后续 `update_sample_state.py` 可以读取 difficulty gain 失败原因，为第三阶段回退做准备。

## 25. 最终判断逻辑

第二阶段完成后，候选题不能再因为“生成出来了且复杂度校验通过”就进入后续流程。

新的判断应该是：

```text
候选题是否合法？
  是 → validate_evolved_question.passed
候选题是否真的有难度收益？
  是 → difficulty_gain_validation.passed
候选题是否没有泄漏答案线索？
  是 → no high-risk leakage tags
候选题是否制造真实竞争判断？
  是 → competitive_judgment_score 达标
候选题是否仍然可回答？
  是 → answerability_score 达标
候选题是否不是靠格式复杂度变难？
  是 → format_complexity_score 达标
```

只有同时满足这些条件，候选题才允许进入：

```text
candidate_selection.py → collect_answers.py → gen_rubric.py → scoring.py
```

这一步完成后，系统会从“候选题生成后直接采纳”升级为“候选题采纳前必须证明有难度收益”的闭环。
