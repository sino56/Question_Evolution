# PoliceQA 题目难度进化项目概况

本目录包含一套用于 **PoliceQA 数据难度进化** 的脚本。

## 核心目标

对当前候选模型（如 Qwen）得分过高的题目进行"题目进化"，使其能更好地区分强模型与弱模型，从而提升合成训练数据的质量。

## 背景

在 PoliceQA 上，rubric 增强遇到瓶颈——候选答案（Qwen）并不显著差于参考答案（GPT），差异多在风格偏好。因此我们把进化对象从 rubric 转向 question：让题目本身变得更难、更需要深度推理。

---

# 1. 整体流程

当前实现是"跨轮链式主流程 + 单轮多候选探索 + 候选选优"，不是完整的多层树搜索。

- 每轮可以为一条样本使用主算子和备选算子生成多个候选，但最终只保留一条主链题目。
- 当样本形成有效边界、被判定不值得继续进化，或进入其他终止状态后，后续轮次会透传该样本并复用已有 reference、rubric 和 scoring_result。

## 1.0 流程总览

```text
Stage 0: data/data.jsonl

Stage 1: scoring.py -> round_0/scored.jsonl

Stage 2: profile_samples.py -> select_evolution_candidates.py

Stage 3: operator_router.py -> question_evolution.py

Stage 4: validate_evolved_question.py -> candidate_selection.py

Stage 5: collect_answers.py -> gen_rubric.py -> scoring.py

Stage 6: analyze_evolution_effect.py -> update_sample_state.py
```

从 **Round 1** 开始，每轮按以下步骤执行：

| 步骤 | 脚本 | 产物 |
| --- | --- | --- |
| 0 | 复制上一轮 scored/state 输入 | round_N/input.jsonl |
| 1 | profile_samples.py | profiled.jsonl |
| 2 | select_evolution_candidates.py | profiled_candidates.jsonl |
| 3 | operator_router.py | routed.jsonl |
| 4 | question_evolution.py | candidates.jsonl |
| 5 | validate_evolved_question.py | validated_candidates.jsonl |
| 6 | candidate_selection.py | evolved.jsonl |
| 7 | collect_answers.py | with_answers.jsonl |
| 8 | gen_rubric.py | rubric.jsonl |
| 9 | scoring.py | scored.jsonl |
| 10 | analyze_evolution_effect.py | effect_analysis.jsonl, effect_matrix.jsonl |
| 11 | update_sample_state.py | state_updated.jsonl, memory bank |

> `question_evolution.py` 的 legacy 单脚本路径仍可用于兼容旧数据或局部调试，但不再是推荐主流程。推荐路径必须经过画像、分流、路由、复杂度/可回答性校验、候选选择、效果统计和状态更新。

## 1.2 文件解释

### Stage 0 与 Round 0

#### `data/data.jsonl`

- 原始输入题库。每行是一道题，通常包含 `prompt`、`meta_info.references`、`rubric`、`score_prompt` 等字段。
- 它表示"当前尚未进入 question evolution 闭环的种子样本"。

#### `scoring.py`

- baseline 评分脚本。
- 先让候选模型回答原题，再让 judge 模型依据现有 rubric 和 score_prompt 打分。
- 作用是得到当前题目对候选模型到底有多难，为后续"是否要进化"提供依据。

#### `round_0/scored.jsonl`

- baseline 评分结果。
- 核心新增字段是 `scoring_result`，里面有候选答案、逐项得分、总分、评分原始输出等。
- 这是后续各轮判断样本是否"高分虚高"或"低分但值得重构"的起点。

### Round N：样本画像与候选分流

#### `round_N/input.jsonl`

- 第 N 轮的输入文件。
- 通常来自上一轮的 `scored.jsonl` 或 `state_updated.jsonl`，相当于把上一轮结果作为这一轮起点继续迭代。

#### `profile_samples.py`

- 对每个样本做"画像 + 诊断"。
- 不是简单打标签，而是生成两类信息：
  - `sample_profile`：这道题主要在考什么能力、属于什么问题形态、期望什么回答模式。
  - `overscore_diagnosis`：如果候选模型分数高，为什么这种高分值得怀疑；如果低分，是否体现了真实能力边界。

#### `profiled.jsonl`

- 在上一轮样本基础上，新增 `sample_profile` 和 `overscore_diagnosis` 后的文件。
- 它是后续"是否继续改题"的直接输入。

#### `select_evolution_candidates.py`

- 根据 `score_rate`、画像和诊断结果，为每个样本分配 `evolution_action`。
- 共五类：
  - `evolve_high_score_overscore`：高分且怀疑是虚高，应该继续进化。
  - `reconstruct_low_score_boundary`：低分但体现了真实边界，可以做边界重构。
  - `probe_middle_score_boundary`：中间分样本存在可进化边界，或上一轮状态要求继续换算子探索。
  - `pass_through_or_scoring_noise`：不值得改题，先透传。
  - `stop_evolution`：这个样本已经应该停止进化。
- 默认高分阈值为 `0.8`，低分阈值为 `0.6`。
- `evolution_state.stop_status` 和 `recommended_next_methods` 的优先级高于当前分数区间。

#### `profiled_candidates.jsonl`

- 在 `profiled.jsonl` 基础上新增 `evolution_action` 和原因说明后的文件。
- 它决定后面样本是进入改题主链，还是直接透传/停止。

### Round N：算子路由、候选生成、复杂度校验

#### `operator_router.py`

- 为"需要进化"的样本选择最合适的改题算子（operator）。
- 它本身不生成题，而是根据以下信息给出 `primary`、`backup`、`avoid` 等 operator 路由结果：
  - 当前样本的 `sample_profile`
  - 当前样本的 `overscore_diagnosis`
  - 上一轮留下的 `evolution_state`
  - memory bank 中的历史成功/失败经验

当前注册算子为 **O10-O18**：

| 算子 ID | 名称 |
| --- | --- |
| O10 | evidence_sufficiency_ladder |
| O11 | unobserved_state_attribution |
| O12 | conjunctive_necessity |
| O13 | minimal_disqualifier |
| O14 | information_closure |
| O15 | counterfactual_threshold_shift |
| O16 | close_alternative_normalization |
| O17 | action_vs_fact_threshold |
| O18 | baseline_scope_mismatch |

- 路由优先级大致为：上一轮推荐方法 > 相似样本成功记忆 > 诊断规则；失败记忆、回滚状态和特定防重复规则会加入 `avoid_operators`。

#### `routed.jsonl`

- 已经完成 operator 路由的文件。
- 可以把它理解为"这一题下一步准备用哪种方法改写"的施工单。

#### `operator_memory_bank.jsonl`

- 正向经验库。
- 记录"某类样本画像 + 某个 operator"曾经打出过有效边界的经验，供后续 router 复用。

#### `failure_memory_bank.jsonl`

- 负向经验库。
- 记录某个 operator 在某类样本上没有打出效果，供后续 router 避免重复踩坑。

#### `question_evolution.py`

- 真正执行改题生成。
- 根据 router 选中的 operator，把原题改写成 1-N 个候选新题。
- 这里生成的是"候选集合"，不是最终定稿。
- 模型/API 调用异常默认重试 3 次，即最多调用 4 次。
- 候选未通过独立校验时，会携带 `reject_reason` 使用同一个 operator 再生成一次；`--validation-retries` 当前只允许 `0` 或 `1`。
- validate-retry 不会自动切换 operator；主/备 operator 的切换发生在多候选生成或下一轮路由中。

#### `candidates.jsonl`

- 候选进化题集合。
- 同一个原始样本可能对应多个 `candidate_id`，代表局部多候选探索。

#### `validate_evolved_question.py`

- 对候选题做独立质量校验。
- 核心检查：
  - 是否单主轴
  - 是否可回答
  - 是否需要题干外知识
  - 是否只是格式复杂度在制造难度
  - 是否重复上一轮题型

#### `validated_candidates.jsonl`

- 候选题加上 `validation_result` 后的文件。
- 每个候选会标明 `passed`、`reject_reason`、`invalid_type` 等信息。

#### `candidate_selection.py`

- 从多个候选中选出一个主链题。
- 它会综合考虑主轴是否唯一、题长是否可控、任务数量、选项数量、格式风险、外部知识风险、重复题型风险等。
- 如果一个样本下的所有候选都不合格，则会回退到原题透传。
- 回退时会恢复 `prompt_old`、`stale_rubric`、`stale_score_prompt` 和 `stale_scoring_result`，并设置 `question_evolved=false`。

#### `evolved.jsonl`

- 每个样本在这一轮最终选中的题目版本。
- 到这里题目才算正式进入下一步闭环，不再是候选草稿。

### Round N：重新生成答案、Rubric 并评分

#### `collect_answers.py`

- 只为 `question_evolved=true` 的题目生成新的参考答案。
- 因为题面已经变了，旧的 `meta_info.references` 不一定还能完整回答新题，所以需要重新采一份强模型答案。
- `question_evolved=false` 的样本原样透传，继续使用已有 reference。

#### `with_answers.jsonl`

- 在 `evolved.jsonl` 基础上补入新参考答案后的文件。
- 它是下一步生成新 rubric 的输入。

#### `gen_rubric.py`

- 根据"当前题目 + 当前参考答案"重新生成 rubric 和 score prompt。
- 作用是保证评分标准和新题目严格对齐，而不是继续沿用旧题的 rubric。
- `question_evolved=false` 的样本不会重新生成 rubric。

#### `rubric.jsonl`

- 带有新 rubric、`rubric_thought_process`、`score_prompt` 的文件。
- 它是重新评分前的完整待评样本。

#### `scoring.py`

- 在这里进入闭环第二次调用。
- 让候选模型回答新题，再用新的 rubric 打分，用来检验本轮改题是否真的把候选模型压到了更清晰的能力边界上。
- `question_evolved=false` 的样本不会重新答题或评分，而是复用上一轮 `scoring_result` 并补齐 `score_rate`。

#### `scored.jsonl`

- 本轮闭环后的评分结果。
- 它既是本轮效果评估的输入，也是下一轮迭代的主要起点。

### Round N：效果统计、状态更新与记忆沉淀

#### `analyze_evolution_effect.py`

- 对比进化前后得分率变化，并结合 `expected_evaluation_focus` 判断这次是否击中了预期错误方向。
- 它会给样本打上 `effect_label`，例如：
  - `effective_boundary_probe`
  - `full_score_no_drop`
  - `no_clear_effect`
  - `score_increased`
  - `repeated_pattern`
  - `needs_manual_review`
  - `invalid_complexity`
  - `pass_through`
- 默认阈值：
  - 满分阈值：`0.99`
  - 明确降分阈值：`0.15`
  - 人工复核降分阈值：`0.05`
  - 分数升高阈值：`0.05`

#### `effect_analysis.jsonl`

- 逐样本的效果分析结果。
- 它是后续更新 `evolution_state` 和写入 memory bank 的直接依据。

#### `effect_matrix.jsonl`

- 聚合后的效果矩阵。
- 更偏统计视角，用于分析不同 operator、不同效果标签之间的分布关系。

#### `update_sample_state.py`

- 为每个样本更新跨轮状态 `evolution_state`。
- 同时根据效果结果写入三类记忆：
  - 成功经验写入 `operator_memory_bank.jsonl`
  - 失败经验写入 `failure_memory_bank.jsonl`
  - 无效候选写入 `invalid_generation_cases.jsonl`

#### `state_updated.jsonl`

- 带最新 `evolution_state` 的样本文件。
- 典型信息包括：
  - 上一轮用了哪个 operator
  - 上一轮得分率是多少
  - 连续满分次数
  - 连续重复 operator 次数
  - 推荐下一轮尝试哪些方法
  - 是否已经到达停止状态
  - 是否需要回滚到上一题目版本

#### `invalid_generation_cases.jsonl`

- 无效候选案例库。
- 记录"生成出来但不该进入主链"的候选题，如不可回答、依赖外部知识、格式复杂度过高、重复题型等。
- 这些记录主要用于后续排查和 router 避坑。

### 终止输出

#### `final/final_scored.jsonl`

- 循环结束时由最后一轮有效输入复制得到。
- 全局提前停止条件包括：
  - 回滚后有效平均得分率低于 `EARLY_STOP_RATE`
  - 连续 `NO_INFO_STOP_ROUNDS` 轮没有有效命中、没有可继续状态，且平均分变化小于 `NO_INFO_MIN_DELTA`
  - 达到 `MAX_ROUNDS`
- 可以理解为整个 question evolution 流水线的收官产物。

# 2. 单样本状态机：进化有效或失败后会发生什么

## 2.1 进化有效

满足以下条件时，效果可被标记为 `effective_boundary_probe`：

1. `question_evolved=true`
2. 候选通过复杂度和可回答性校验
3. 题型没有与上一轮重复
4. 分数下降达到明确降分阈值，默认至少为 `0.15`
5. `candidate_answer` 的错误方向与 `expected_evaluation_focus` 匹配
6. 命中置信度达到 `medium` 或 `high`

> 下降至少 `0.05`、打破原来的满分或缺少 focus 匹配时，可能进入 `needs_manual_review`，但不会自动标记为 `effective_boundary_probe`。默认下降至少 `0.30` 且 focus 匹配、候选答案存在时，命中置信度为 `high`。

有效后：

- 当前进化题保留，不回滚。
- `evolution_state.stop_status` 设置为 `effective_boundary_sample`。
- 清空 `recommended_next_methods`，该样本不再继续改题。
- 成功经验写入 `operator_memory_bank.jsonl`，供相似样本后续路由使用。
- 下一轮 `select_evolution_candidates.py` 会输出 `stop_evolution`，后续步骤原样透传并复用已有评分结果。

## 2.2 候选生成或校验失败

| 失败类型 | 处理动作 |
| --- | --- |
| API、解析或模型返回异常 | 按 `--retries` 重试 |
| 独立校验失败 | 携带 `reject_reason`，使用同一 operator 做一次 validate-retry |
| 某个候选失败但同组仍有合格候选 | 选中其他综合分更高的候选 |
| 全部候选失败 | 回退原题，恢复旧 rubric、score_prompt 和 scoring_result，并把无效候选写入 `invalid_generation_cases.jsonl` |

## 2.3 题目合格但压测效果失败

| effect_label | 下一步 |
| --- | --- |
| `full_score_no_drop` | 第一次通常为 `continue_with_new_operator`；连续满分达到 2 次后为 `local_tree_search_needed`，下一轮增加候选探索 |
| `no_clear_effect` | 保留当前题目版本，推荐新 operator 继续探索 |
| `needs_manual_review` | 保留为低置信候选，下一轮换 operator 或人工复核 |
| `repeated_pattern` | 第一次换 operator；同一 operator 连续重复达到 2 次后进入 `stable_high_score_stop` |
| `score_increased` | 回滚原题、旧 rubric、旧评分和旧 score_rate，记录失败题目，设置 `rollback_and_reroute`，下一轮避开失败 operator |
| `invalid_complexity` | 标记为 `invalid_complexity_sample`；若依赖外部知识或空题面则标记为 `unanswerable_or_trap_sample` |

其中只有 `score_increased` 会在状态更新阶段主动恢复上一版本；`full_score_no_drop`、`no_clear_effect` 和 `needs_manual_review` 通常保留当前题目，再通过下一轮重新路由。

## 2.4 切换算子时基于哪个题目继续进化

当前实现是链式进化，不是通用树回溯。切换 operator 时使用父题还是当前子题，取决于失败类型：

| 场景 | 下一轮使用的题目 | 处理方式 |
| --- | --- | --- |
| `score_increased` | 上一轮父题 | 恢复 `meta_info.prompt_old`、`stale_rubric`、`stale_score_prompt`、`stale_scoring_result` 和原 score rate，再从父题重新路由，避开失败 operator |
| 所有候选均未通过校验 | 上一轮父题 | candidate_selection.py 立即恢复父题并设置 `question_evolved=false`，无效候选写入 `invalid_generation_cases.jsonl` |
| `full_score_no_drop` | 当前进化后的子题 | 不回滚；把当前题写入 `state_updated.jsonl`，下一轮在当前题上更换 operator |
| `no_clear_effect` | 当前进化后的子题 | 不回滚；保留当前题并使用 `recommended_next_methods` 重新路由 |
| `needs_manual_review` | 当前进化后的子题 | 不回滚；保留低置信候选，下一轮换 operator 或等待人工复核 |
| `repeated_pattern` | 通常为当前进化后的子题 | 第一次重复时不回滚并换 operator；同一 operator 连续重复达到 2 次后停止该样本 |
| `effective_boundary_probe` | 当前有效题 | 不再切换 operator；设置为终止状态并在后续轮次透传 |

可以简化为：

```text
生成题不合法 -> 回到父题

进化后分数反而升高 -> 回到父题，切换 operator

题目合法但压测效果不清晰 -> 留在当前子题，切换 operator

进化形成有效边界 -> 保留当前题并停止继续进化
```

这里的"父题"是当前题目的直接上一版本，不一定是最初的根题。每次进化只通过 `meta_info.prompt_old` 和 `meta_info.stale_*` 保存一层父节点；当前代码没有实现向多层祖先回溯、遍历历史兄弟分支或从根题重新搜索。

# 3. 当前版本较之前版本的差异

V3 最主要的修改：

- O1-O9 替换为 O10-O18。
- 重写 question evolution prompt。
- 从"显式找缺口、拆层级"转向"接近业务判断竞争"。
- 强化题面防泄漏、干扰项接近度和单变量变化规则。

流程行为也有几处实质变化：

## 3.1 中分样本现在可以继续进化

- V2 通常透传。
- V3 新增 `probe_middle_score_boundary`。

## 3.2 分数升高的处理完全不同

- V2：停止该样本。
- V3：回滚父题、避开失败算子、重新路由。

## 3.3 连续满分不再过早停止

- V2：换算子后仍满分可能直接停止。
- V3：进入 `local_tree_search_needed`，增加候选探索。

## 3.4 全局停止判断更准确

V3 使用回滚后的 `state_updated.jsonl` 计算平均分，并检查是否还有可继续进化的样本。

## 3.5 算子效果归因被修正

V3 按最终选中的候选算子记录成功或失败，不再容易使用过期 metadata。
