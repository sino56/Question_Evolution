# 实验迭代速度优化方案

- **文档版本**：v3.0
- **修订日期**：2026-07-17
- **项目**：Question_Evolution / PoliceQA 题目难度进化
- **读者**：维护实验流水线、处理阶段中断和调整模型调用并发的内部工程师。
- **读后可执行的动作**：根据实验规模和模型服务额度配置并发，在阶段中断后识别可恢复产物，并从阶段性能事件定位本地瓶颈或服务端限流。

## 1. 优化目标与不变量

该轮优化的目标是缩短本地处理、磁盘 I/O 和不必要等待造成的迭代时间，同时让中断后的实验能够从已确认的进度继续执行。优化不以减少模型调用为手段，也不改变题目进化的业务决策。

以下实验语义保持不变：

- 每条样本的进化分流、算子路由、候选选择、回退、效果判定和状态更新规则不变；
- 参考答案采样次数、Qwen 回答次数、Qwen/GPT 评分重复次数、trial/repeat 编号及最终聚合规则不变；
- 远程请求仍使用既定的模型、Prompt、参数、重试和 API key 轮转规则；
- memory bank 仍在轮末依据效果分析追加，下一轮再作为路由输入；
- 透传样本继续复用已有的参考答案、rubric 和评分结果，但会先校验这些结果是否完整可用。

优化的核心原则是将"样本处理并发度"和"模型服务实际在途请求数"分开控制：前者用于限制本地任务和队列规模，后者用于遵守服务额度并降低突发限流。

## 2. 当前优化后流程

编排入口仍按轮次串行提交业务阶段。每个阶段只有在上游正式产物已发布后才读取该产物，因此并发只存在于阶段内部，不会改变轮级顺序或提前写入 memory。

```
已准入输入 JSONL
        |
        v
Round 0：发布 input 产物
        |
        v
评分阶段
流式读取 -> 有界样本 worker -> Qwen / GPT 公平请求池 -> 批量发布 scored.jsonl
        |
        v
Round N：发布上一轮 state_updated/scored 输入
        |
        v
画像阶段
流式读取 -> 有界 worker -> 压缩原始响应 trace -> 发布 profiled.jsonl
        |
        v
候选分流 -> 算子路由
                 |
                 +--> 只读 memory 索引与匹配缓存
                 |
                 v
题目进化
有界 worker -> 候选组检查点 -> 压缩原始响应 trace -> 发布 candidates.jsonl
        |
        v
复杂度/可回答性校验
复用同规则版本的上游本地校验结果；不匹配时重新计算
        |
        v
候选选择 / 原题透传
        |
        v
参考答案采集
流式读取 -> 有界样本 worker -> 公平全局请求池 -> 发布 with_answers.jsonl
        |
        v
Rubric 生成
有界 worker -> 发布 rubric.jsonl
        |
        v
评分阶段
流式读取 -> 有界样本 worker -> Qwen / GPT 公平请求池
          -> 压缩 judge trace -> 发布 scored.jsonl
        |
        v
效果分析 -> 状态更新 -> 追加 memory bank -> state_updated.jsonl
        |
        +--> 继续下一轮，或导出最终 scored JSONL
```

图中的"发布"统一表示：阶段先写入临时产物和恢复信息，确认完整后再将其作为下游可见的正式 JSONL。候选选择、效果分析和状态更新等需要全局关联的阶段仍保持原有业务处理方式，但同样通过统一发布机制输出正式产物。

## 3. 阶段产物、批量写入与中断恢复

### 3.1 统一 JSONL 读写

流水线将正常阶段产物统一为 JSONL，并保留对历史 JSON 数组输入的兼容读取。逐行读取时会跳过空行，并在解析失败时报告阶段名、文件和行号。这样可以尽早发现局部脏数据，避免把整个文件读入后才暴露错误；对于画像、答案采集和评分等可逐样本处理的阶段，也可使输入规模不再直接决定待处理任务数。

输出由单一阶段 writer 管理。业务 worker 只负责产生结果，writer 在达到记录数、字节数或时间阈值后批量落盘。其目的不是改变输出内容或业务顺序，而是降低每条记录单独写入、flush 和同步等待带来的 I/O 开销，同时保留可控的恢复粒度。

### 3.2 partial、checkpoint 与正式发布

每个正式 JSONL 都有一组同目录的运行产物：

- partial 文件保存尚未发布的业务记录；
- checkpoint 以稳定记录键记录已持久化的输出位置；候选多输出场景在整个候选组写完后才标记完成；
- manifest 记录输入摘要、配置摘要、正式输出摘要、代码摘要和已登记 sidecar 的校验信息。

写入时，业务记录会先进入 partial 文件，再追加对应 checkpoint。阶段成功后关闭写入句柄、生成 manifest，并以原子替换发布正式 JSONL。这样下游只会看到完整阶段结果，不会把半写入文件当成可用输入。

阶段中断后，只要 partial 和 checkpoint 仍一致，重启时会读取 checkpoint 并跳过已确认的记录或候选组，继续处理未完成部分；正式产物则必须同时通过 manifest 中的记录数和内容摘要校验，才能被编排入口跳过。校验不通过时，编排入口会拒绝静默覆盖，避免在不明状态下把新旧实验结果混合。

### 3.3 性能事件

阶段 writer 会将输入/输出记录数和字节数、解析、计算、序列化、恢复耗时、flush 次数、checkpoint 命中情况、队列深度和 RSS 峰值写入轮次目录的 performance_events.jsonl。这些数据的用途是区分本地解析、排队、写入和远程等待问题，避免将模型服务波动误判为本地优化收益。

## 4. 有界任务调度

可独立处理的远程阶段使用固定数量的 worker、有限输入队列和有限输出队列。reader 只有在队列有容量时才继续提交记录，worker 完成后由单 writer 接收结果并释放容量；需要保持输入顺序的阶段由 writer 按序输出，候选组进化和评分等允许完成顺序的阶段则在结果就绪后写出。

该机制的目的是把内存峰值、协程数量和待写结果限制在并发配置附近，而不是随着输入文件线性增长；同时让首批完成的结果可以尽早进入可恢复的 partial 产物。画像、参考答案采集和评分以流式 reader 接入该机制；题目进化和 rubric 生成以有界 worker 接入，题目进化仍会在开始时读取记录以分配本轮候选预算。

worker 并发度只描述"同时处于该阶段流程的样本数"。它不能替代模型服务限流，也不表示同一时刻可以发送多少 API 请求。

## 5. 评分的受控并行

评分阶段将样本 worker、Qwen 请求池和 GPT 请求池分别配置。默认上限均为 20，但实际值应根据所用服务的并发、每分钟请求数和 token 限制下调。

```
评分样本 worker
        |
        +--> 回答请求 ------> Qwen 公平请求池
        |
        +--> Qwen 评分 -----> 同一个 Qwen 公平请求池
        |
        +--> GPT 实验评分 --> GPT 公平请求池
```

Qwen 回答和 Qwen judge 共享同一个请求池，因此两类请求的在途总数不会因分别配置而叠加；GPT judge 使用独立服务池。每次真实网络调用才申请名额，请求成功或异常后立即归还，重试等待不占用名额，下一次重试重新申请。这样既保留原有重试语义，又不会让退避中的任务阻塞其他样本。

在一个样本内部，独立的回答 trial 可以并行，已获得回答的独立评分也可以并行；Qwen 和 GPT 评分可在各自依赖满足后并行。结果在汇总前按 trial_index 和 repeat_index 归位，所以完成先后不会改变试次含义、结果列表顺序或评分聚合。

请求池按样本公平分配名额：多个活跃样本存在时，优先让不同样本各获得一个名额，再使用剩余容量处理同一样本的后续 trial 或 repeat。优化目的是减少少数慢题或多次评分的题目独占全部额度造成的尾部等待，并保证任意路径下的服务并发受限。

## 6. 参考答案采集的受控调度

参考答案采集同样区分样本 worker 并发和全局请求并发。`--concurrency` 控制同时处于采集流程的样本数，`--request-concurrency` 控制参考答案服务的真实在途请求上限，默认不超过 20；每题的采样次数不会再直接放大服务并发。

每次采样请求按所属样本进入公平请求池。多个样本同时采集时，调度器优先为不同样本分配请求名额，再在服务有余量时执行同题的后续采样。质量校验、key 切换和重试退避位于请求名额之外。这样可以避免"样本并发 x 每题采样数"形成突发流量，降低 429、空响应和重试堆积带来的有效吞吐下降，而不影响采样次数、答案质量校验和未进化样本的透传规则。

## 7. 原始响应 trace 分层

画像、题目进化和评分阶段将原始模型响应从主业务记录中移至压缩的 trace sidecar。主 JSONL 保留业务字段和 trace 标识，sidecar 保存记录键、阶段、内容摘要、编码和原始文本；发布 manifest 会登记 sidecar 的校验信息。

这种分层的目的，是让下游阶段不必在每次读取、复制和序列化业务记录时携带冗长原始响应，同时仍能通过 trace 标识追溯审计内容。评分阶段会分别外置代表性 judge 原始响应和各 trial/repeat 的 judge 原始响应，因而不会改变评分结果或丢失诊断依据。

## 8. memory 路由索引与本地校验复用

### 8.1 memory 匹配

算子路由在加载正、负 memory bank 后，基于样本画像和过分原因的 signature 建立只读倒排索引，并为相同 signature 缓存匹配结果。路由时先通过索引取出可能命中的 memory，再沿用原有相似度计算和排序规则生成 primary、backup、avoid 及理由。

索引和缓存只改变"从哪些记录开始比较"的本地查找路径，不修改 memory 的追加顺序、保留内容或路由判定规则。其目的在于避免每条样本重复扫描整个 memory bank，使 bank 随轮次增大时的匹配开销保持可控。

### 8.2 本地复杂度校验结果复用

题目进化阶段会保存其本地复杂度/可回答性校验结果及规则版本摘要。正式校验阶段发现该结果的规则版本与当前版本一致时，直接复用结果；缺失或版本不一致时仍按当前规则重新计算。启用独立 LLM 校验时，LLM 校验照常执行并与本地结果合并。

复用的目的，是消除同一候选在相同规则下的重复本地计算，同时利用规则版本摘要防止旧结果掩盖规则变更。该机制不会跳过需要远程模型参与的独立校验，也不会改变候选选择所依据的校验字段。

服务出现限流、超时或空响应时，优先参考日志中的请求池峰值与 performance_events.jsonl：若在途峰值接近服务安全值，应降低对应 request concurrency；若请求池未满而队列、解析或写入时间高，则再检查本地 worker 配置和产物体积。不要通过提升样本 worker 数绕过请求池，也不要将不同模型服务的额度混为一个数值。

## 10. 已有验证

项目测试已覆盖正式产物的原子发布与 manifest 校验、checkpoint 恢复、trace 可追溯性及业务字段等价比较；同时覆盖评分和答案采集的全局请求上限、跨样本公平调度、样本内并行、重试释放名额、连接生命周期以及透传复用。运行全量回归可使用：

```bash
pytest -q
```

执行真实实验时，除了确认测试通过，还应检查本轮所有正式产物均有匹配 manifest、失败产物已被处理、请求池峰值未超过配置上限，并结合性能事件判断本地优化是否实际改善了当前数据规模下的迭代时间。
